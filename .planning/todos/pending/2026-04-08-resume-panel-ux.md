## Problem

실사용 테스트 중 다음 이슈가 반복적으로 드러났다.

1. 이력서 자료를 "없음"으로 시작했을 때 직군 질문이 개발 직군 위주로 수렴한다. 채팅 이력 기반 개인화라기보다 프롬프트/예시 편향 가능성이 크며, 비개발 직군 사용자를 배제하는 인상을 준다.
2. 회사명/기간 입력 후 리서처 결과를 기다리는 동안 약 2분 가까이 멈춘 것처럼 보이는 구간이 있다. 사용자는 입력 반영 여부와 남은 시간을 알 수 없고, 대기 중에도 수집 가능한 정보가 놀고 있다.
3. 회사 리서치에서 토큰 소모가 크다. 현재 구조는 경력 회사별 병렬 리서처 + 타겟 회사 리서처를 동시에 돌리고, 회사 기본/서비스/기술/조직/최근 동향/경쟁사/JD까지 풀스캔하며, WebSearch/WebFetch 실패 시 Playwright snapshot fallback까지 사용한다. 이후 이 결과가 다른 에이전트와 프로파일러에 반복 주입되어 누적 비용이 커진다.
4. AskUserQuestion 셀렉트 박스에서 빈 영역으로 포커스가 간 상태에서 Enter를 누르면 질문이 종료되는 케이스가 있다. 이 경우 빈 응답처럼 처리되어 텍스트 폴백으로 내려가고, 동일 질문이 셀렉트 UI로 복구되지 않는다.
5. 서브에이전트는 가능한 한 퓨어한 컨텍스트로 보내고 싶다. 현재는 메인 대화 히스토리를 통째로 포크하지 않더라도, 런타임 전역 지침/툴/MCP/플러그인 환경이 남아 있어 컨텍스트 노이즈 우려가 있다. 오케스트레이터가 실제 업무 범위보다 큰 컨텍스트를 반복 주입하는 구조도 토큰 낭비와 품질 저하 가능성이 있다.
6. 질문 UI가 단일 셀렉트 중심이라, 실제로는 복수 선택이 자연스러운 상황도 하나만 고르게 만든다. 사용자는 체크박스처럼 여러 항목을 동시에 표시하고 제출하고 싶어 한다.
7. 질문을 한 번에 하나씩만 던져서 인터뷰 흐름이 길어진다. 서로 연관된 질문은 여러 개를 탭 단위로 묶어 답하게 하고, 마지막에 "내가 고른 답이 맞는지" 한 번 더 확인하는 리뷰 UX가 필요하다.
8. 리서처 조사 결과가 메인 세션에서 영속적으로 모이는 저장소가 없어, 결과를 그때그때 받아 바로 소비하는 구조에 가깝다. 초기 대기 시간을 줄이려면 조사 결과를 유의미한 단위로 즉시 저장하고, 메인/프로파일러가 점진적으로 읽어 활용할 수 있어야 한다.

## Solution

다음 방향으로 정리해서 후속 작업한다.

1. 라운드 0 직군 수집 재설계
   - 이력서 없음 + JD 미확정 상태에서는 JD 기반 직군 추정 대신 범용 직군 카테고리 + 자유입력으로 먼저 받는다.
   - 개발자 예시만 강하게 보이는 프롬프트/에이전트 예시를 줄이고, 비개발 직군 예시 비중을 늘린다.
2. 리서치 대기 구간 UX 개선
   - 리서처는 백그라운드에서 돌리고, 앞단에서는 직군/타겟 포지션/강조 회사/현재 재직 여부 같은 비의존 질문을 계속 진행한다.
   - 진행률, 예상 대기 시간, 현재 처리 단계 같은 상태를 보여주고 "기다리면서 계속 진행" 선택지를 제공한다.
3. 리서처 경량화 + 토큰 절감
   - 과거 회사는 경량 조사, 타겟 회사/JD만 심화 조사로 분리한다.
   - 첫 질문 생성에 필요한 최소 팩트만 먼저 수집하고, 경쟁사/최근 동향/팀 구조 등은 필요 시 추가 조사한다.
   - Playwright fallback의 페이지 수와 snapshot 횟수에 하드캡을 둔다.
   - 리서처 결과는 다운스트림 에이전트에 원문 전체가 아니라 압축 요약 형태로 전달하고, 같은 회사/JD는 세션 내 캐시한다.
4. 셀렉트 박스 복원력 강화
   - 선택지 미선택 상태의 Enter는 무시한다.
   - 빈 포커스 Enter를 빈 응답/실패로 취급하지 않는다.
   - AskUserQuestion이 비정상 종료되더라도 같은 질문을 셀렉트 박스로 즉시 재오픈한다.
   - 텍스트 폴백은 실제 툴 실패/타임아웃일 때만 허용한다.
5. 서브에이전트 최소 컨텍스트 모드 정리
   - 메인 오케스트레이터는 서브에이전트에 작업 범위, 필요한 파일, 압축 요약만 전달하는 패턴으로 표준화한다.
   - 대화 히스토리 포크가 불필요한 경우 항상 최소 컨텍스트 호출 방식을 우선한다.
   - 리서처/프로파일러/질문 생성 에이전트에 전달하는 컨텍스트를 "필수 사실 요약" 단위로 재설계한다.
6. 질문 입력 컴포넌트 확장 검토
   - 단일 선택 외에 다중 선택이 필요한 질문 유형을 정의하고, 체크박스 형태의 멀티 셀렉트 UX를 제공한다.
   - 멀티 셀렉트 결과는 에피소드 추출 단계에서 해석 가능하도록 저장 포맷과 후속 질문 규칙을 함께 설계한다.
7. 배치 질문 + 최종 확인 UX
   - 독립적인 짧은 질문들은 여러 탭/스텝으로 묶어서 한 세트로 제시한다.
   - 세트 마지막에 사용자가 선택한 답변 요약을 보여주고, 수정/확정 단계를 제공한다.
   - 이 흐름은 단순 정보 수집 라운드와 리서처 대기 구간에서 우선 적용 후보로 검토한다.
8. 리서처 결과 캐시 + 점진적 프로파일링
   - `.resume-panel/research-cache.json` 또는 `research-inbox.jsonl` 같은 저장소를 두고, 회사별/타겟 JD별 조사 결과를 유의미한 단위로 즉시 append 또는 merge 저장한다.
   - 리서처는 "조사 완료 후 한 번에 전달"이 아니라, 핵심 팩트가 확보될 때마다 캐시를 갱신한다.
   - 메인 오케스트레이터는 대기 중에도 이 캐시를 읽어 질문 생성에 바로 활용하고, 프로파일러도 신규 조사 결과가 들어올 때마다 증분 분석할 수 있게 한다.
   - 기존 `findings-inbox -> hook -> additionalContext` 패턴과 유사하게 설계하되, 조사 캐시는 findings와 분리하여 원천 팩트 저장소 역할을 맡긴다.
   - 최소 스키마 예시:
     - `company_key`, `source_type`(career_company|target_jd), `facts`, `confidence`, `updated_at`, `ready_for_questions`, `ready_for_profiler`
   - 초기 목표는 "첫 질문 시작 전 모든 조사 완료"를 요구하지 않고, 질문 생성에 필요한 최소 팩트만 확보되면 인터뷰를 시작하는 것이다.
