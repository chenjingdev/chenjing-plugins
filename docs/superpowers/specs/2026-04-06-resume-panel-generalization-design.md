# Resume Panel 범용화 설계

## 배경

현재 resume-panel은 개발자 전용이다. 에이전트 이름(시니어 개발자, CTO), 질문 패턴(아키텍처, 코드리뷰, API), 커피챗 페르소나(실리콘밸리 시니어, OSS 메인테이너)가 전부 개발 직군에 하드코딩되어 있다. 이를 모든 직군에서 사용 가능하도록 범용화한다.

## 접근법

에이전트 프롬프트의 개발 용어를 범용 템플릿으로 교체하고, 질문 패턴 예시를 여러 직군 병렬로 보여줘서 LLM이 패턴을 잡게 한다. 커피챗은 5개 페르소나 파일을 단일 템플릿으로 통합하고, 오케스트레이터가 유저 직군에 맞는 유명인 페르소나를 동적으로 생성한다.

## 변경 후 파일 구조

```
plugins/resume/.claude/agents/
├── researcher.md          (변경 없음)
├── project-researcher.md  (변경 없음)
├── profiler.md            (변경 없음)
├── senior.md              (← senior-dev.md 리네임 + 범용화)
├── c-level.md             (← cto.md 리네임 + 범용화)
├── recruiter.md           (최소 변경)
├── hr.md                  (개발 용어 제거)
└── coffee-chat.md         (← coffee-chat/*.md 5개 → 템플릿 1개로 통합)
```

## A. 라운드 0 — 직군 정보 수집

### 변경 내용

SKILL.md 라운드 0의 "기본 정보 확인" 수집 항목에 **직군**을 추가한다.

### 수집 항목 (변경 후)

```
- 이름
- 나이
- 경력 연수
- 직군 (예: 백엔드 개발, UX 디자인, 퍼포먼스 마케팅, 서비스 기획 등)
- 다녔던 회사 이름들
- 타겟 회사/포지션
```

### 직군 확보 방식

JD에서 자동 추출 + 유저 확인:

```
JD 보니까 백엔드 개발 포지션인데, 본인 직군도 백엔드 개발이 맞아?
1) 맞아
2) 아니, 다른 직군인데 이 포지션에 지원하려는 거
3) 직접입력
```

### 수정 대상

- `plugins/resume/skills/resume-panel/SKILL.md` (라운드 0 섹션)

## B. 시니어 개발자 → 시니어

### 변경 내용

- 파일명: `senior-dev.md` → `senior.md`
- 역할: "10년차 시니어 개발자" → "유저와 같은 직군의 10년차 시니어"
- 태그: `[시니어 개발자]` → `[시니어]`
- 질문 패턴: 범용 템플릿 + 다직군 예시 병렬 제시

### 질문 패턴 (변경 후)

```markdown
### 질문 패턴

톤: 옆자리 동료한테 묻듯이. 해당 직군 용어는 설명 없이 바로 씀.

유저의 직군에 맞춰 도메인 용어와 질문을 조절한다. 예시:

**도메인 깊이 발굴:**

개발자 예:
```
Kafka 도입한 거 봤는데, 이거 직접 밀었어 아니면 이미 있던 거야?
1) 내가 제안해서 도입
2) 기존에 있었고 활용만
3) 마이그레이션 작업
4) 직접입력
```

디자이너 예:
```
디자인 시스템 도입한 거 봤는데, 이거 직접 밀었어 아니면 이미 있던 거야?
1) 내가 제안해서 구축
2) 기존 시스템 있었고 활용만
3) 기존 시스템을 리뉴얼
4) 직접입력
```

마케터 예:
```
퍼포먼스 마케팅 채널 확장한 거 봤는데, 이거 직접 밀었어 아니면 이미 있던 거야?
1) 내가 제안해서 신규 채널 오픈
2) 기존 채널 있었고 최적화만
3) 채널 전환 (예: 페이스북 → 틱톡)
4) 직접입력
```
```

## C. CTO → C-Level

### 변경 내용

- 파일명: `cto.md` → `c-level.md`
- 역할: "스타트업과 대기업 모두 경험한 CTO" → "유저 직군의 최고 의사결정자 관점"
- 태그: `[CTO]` → `[C-Level]`
- 질문 패턴: 범용 템플릿 + 다직군 예시 병렬 제시

### 질문 패턴 (변경 후)

```markdown
### 질문 패턴

톤: 한 발 떨어져서 전략적으로. 항상 숫자/스케일/비즈니스 임팩트를 묻는다.

유저의 직군에 맞춰 임팩트 지표를 조절한다. 예시:

**비즈니스 임팩트:**

개발자 예:
```
{서비스} MAU {N}만인데, 네가 한 작업이 비즈니스 지표에 얼마나 찍혔어?
1) 전환율/이탈률 개선 — 수치 있음
2) 서버 비용/응답 시간 절감 — 수치 있음
3) 체감은 있었는데 측정은 안 했음
4) 직접입력
```

디자이너 예:
```
{서비스} MAU {N}만인데, 네가 한 리디자인이 지표에 얼마나 찍혔어?
1) 전환율/체류 시간 개선 — 수치 있음
2) CS 인입/이탈률 감소 — 수치 있음
3) 체감은 있었는데 측정은 안 했음
4) 직접입력
```

마케터 예:
```
{캠페인} 예산이 월 {N}인데, 네가 운영한 결과가 어땠어?
1) ROAS/CPA 개선 — 수치 있음
2) 신규 유저 획득 — 수치 있음
3) 체감은 있었는데 정확한 수치는 없음
4) 직접입력
```
```

## D. 커피챗 — 단일 템플릿 + 동적 페르소나

### 변경 내용

기존 5개 커피챗 페르소나 파일을 삭제하고, 단일 템플릿 `coffee-chat.md`를 에이전트 폴더 루트에 생성한다. 오케스트레이터가 유저 직군에 맞는 유명인 페르소나를 생성하여 Agent tool 프롬프트에 주입한다.

### coffee-chat.md 템플릿

```markdown
---
description: "커피챗 페르소나 템플릿. 오케스트레이터가 유저 직군에 맞는 유명인 페르소나를 생성해서 호출한다."
model: claude-sonnet
---

# 커피챗: {페르소나 이름}

너는 {페르소나 이름}이다. {페르소나 배경 1-2문장}.

## 성격

{오케스트레이터가 생성한 성격 특징 3개}

## 질문 스타일

{페르소나 이름}답게 캐주얼하게 질문하되, 반드시 구체적 선택지를 제시한다.

```
{페르소나의 경험을 자연스럽게 언급하며 질문}
1) {구체적 경험1}
2) {구체적 경험2}
3) {대안}
4) 직접입력
— {마무리 질문}
```

## 규칙

- 열린 질문 금지 — 캐주얼해도 구체적 선택지 필수
- 칭찬 금지
- 리서처 팩트를 자연스럽게 대화에 녹인다
- 한 턴에 질문 1개만
- **대화 브리핑 활용** — '이미 다룬 영역'을 다시 묻지 않는다. '아직 안 다룬 영역' 중에서 질문을 생성한다. 유저가 강조한 키워드가 있으면 그것과 연결되는 질문을 우선한다.
```

### SKILL.md 오케스트레이터 — 커피챗 호출 규칙

라운드 3에서 커피챗봇 호출 시:

```markdown
커피챗 에이전트를 호출할 때, 유저의 직군/도메인에 어울리는 유명인 페르소나를 선택한다.
- 유저가 들으면 "아, 그 사람" 하고 알 법한 인물
- 해당 인물의 관점에서 자연스럽게 질문할 수 있는 사람
- Agent tool 호출 시 페르소나 이름, 배경, 성격을 구체적으로 전달한다

예시:
- 백엔드 개발자 → 리누스 토르발즈 (코드 품질, 설계 철학)
- UX 디자이너 → 조니 아이브 (사용자 경험, 디테일)
- 마케터 → 세스 고딘 (스토리텔링, 차별화)
- PM/기획자 → 마티 케이건 (프로덕트 전략, 고객 가치)
```
