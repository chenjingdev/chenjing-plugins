# Resume Panel — 이력서 소스 수집 + 작성 스킬 설계

## 개요

전문가 패널 에이전트들이 번갈아 등장하며 유저의 경력 에피소드를 발굴하고, JD 맞춤 이력서를 생성하는 Claude Code 플러그인 스킬.

기존 `resume-source` 스킬을 완전히 대체한다.

## 핵심 원칙

1. **열린 질문 금지** — 모든 질문은 리서처 조사 결과를 포함한 구체적 질문이어야 한다
2. **선택지 필수** — 질문 시 반드시 2~3개 선택지 제시. 유저는 번호 또는 자유 텍스트로 응답
3. **칭찬/감탄 금지** — "대단하네요", "오호!" 없음
4. **팩폭 허용** — 이력이 연차/타겟 대비 부족하면 솔직히 말하고 기준선 제시
5. **한 턴에 질문 1개** — 한꺼번에 몰아치지 않음
6. **메모리 활용** — 이미 아는 정보는 자동 채우고 확인만

### 금지 질문 예시

```
✗ "그래서 어떻게 됐어?"
✗ "오호? 또 다른 경험은?"
✗ "이야 재밌었겠다! 그 다음은?"
✗ "동시접속 1000명이던데 무슨 일 없었어?"
```

### 허용 질문 예시

```
✓ "조사해보니 ㅇㅇ회사에서 ㅁㅁ플랫폼 만들었던데 이거 맞아?"
✓ "ㅁㅁ 플랫폼 동시접속 1000명 이상인데 접속 몰릴 때 어떻게 처리했어?"
✓ "ㅁㅁ에서 CDN 쓰고 있던데 캐싱 전략은 너가 설계한 거야?"
```

차이: 구체적 행동을 묻는 질문 vs 유저가 알아서 떠올리길 바라는 질문.

## 아키텍처

```
유저 ←→ 오케스트레이터(SKILL.md) ←→ 패널 에이전트들(.claude/agents/)
              │                              ├── researcher.md
              │                              ├── senior-dev.md
              │                              ├── cto.md
              │                              ├── project-researcher.md
              │                              ├── profiler.md
              │                              ├── recruiter.md
              │                              ├── hr.md
              │                              └── coffee-chat/*.md (풀)
              │
              ├── resume-source.json  (구조화 데이터)
              └── resume-draft.md     (이력서 초안)
```

- **오케스트레이터(SKILL.md):** 진행 관리, 유저 응답 라우팅, 에이전트 선택 호출, 데이터 축적
- **에이전트들(.claude/agents/):** 각자 독립 컨텍스트에서 실행. 오케스트레이터에게 질문/피드백 리턴. 유저와 직접 대화 안 함.

## 에이전트 상세

### 리서처 (researcher.md)

- **역할:** 정보 수집 전담. 유저와 직접 대화 안 함.
- **조사 대상 2가지:**
  - 유저 경력 회사: 규모, MAU, 매출, 펀딩, 팀 구조, 기술스택, 기술블로그, 경쟁사
  - 타겟 회사/포지션: JD 분석, 채용 기준, 기술스택, 팀 구조
- **실행 방식:** 회사당 1인스턴스 병렬 실행. 에이전트 파일은 1개, 실행은 N개.
- **도구:** WebSearch, WebFetch, Playwright MCP (JS 렌더링이 필요한 JD 페이지, 기술 블로그 등)
- **Playwright 실행 규칙:**
  - 헤드리스 모드 필수 — 유저 화면에 브라우저 띄우지 않음
  - 리서처 인스턴스별 별도 브라우저 세션 — 에이전트 간 세션 겹침 방지
- **산출:** 조사 결과를 오케스트레이터에게 리턴 → 다른 에이전트 질문에 주입

### 프로젝트 리서처 (project-researcher.md)

- **역할:** 유저의 개인 프로젝트 관련 로컬 채팅 세션(codex, claude code, open code 등)을 탐색하여 프로젝트 정보를 체계적으로 정리. 유저와 직접 대화 안 함.
- **트리거:** 유저가 개인/사이드 프로젝트를 언급했을 때
- **내부 Map-Reduce 파이프라인:**
  1. **Collector** (1개) — 로컬 채팅 세션 디렉토리 탐색, 키워드로 관련 세션 파일 검색 (Glob/Grep). 산출: 관련 세션 파일 경로 목록.
  2. **Cleaner** (세션당 1개, 병렬) — 원본 대화 이력에서 노이즈 제거. 제거 대상: tool call/응답, MCP 메시지, 터미널 출력, 시스템 프롬프트, 파일 내용 덤프 등. 산출: `유저: ... / 챗봇: ...` 형태의 정제된 대화문.
  3. **Extractor** (세션당 1개, 병렬) — 정제된 대화문 1개만 읽음. 추출: 기술 결정, 문제 해결, 아키텍처, 사용 기술, 결과물. 산출: 세션별 구조화된 인사이트 문서.
  4. **Synthesizer** (1개) — 모든 인사이트 문서 취합, 중복 제거, 패턴 발견, 테마별 정리. 산출: 통합 인사이트 모음.
  5. **Summarizer** (1개) — 통합 인사이트 → 최종 프로젝트 요약. 산출: 프로젝트 정리 문서 (→ 프로파일러/패널 에이전트에 공급).
- **도구:** Glob, Grep, Read
- **설계 이유:** 대화 이력은 양이 많고 대화문 형식이라 단일 에이전트가 전부 읽으면 컨텍스트 초과. Stage 2(Cleaner)에서 노이즈를 먼저 제거하고, Stage 3(Extractor)에서 인사이트를 추출. 둘 다 세션당 병렬 처리. Stage 4~5는 이미 압축된 인사이트만 받아 가볍게 처리.

### 프로파일러 (profiler.md)

- **역할:** 유저에 대한 모든 시그널을 종합하여 프로파일링. 유저와 직접 대화 안 함.
- **입력 시그널:**
  - 유저 직접 답변 (라운드 0 기본 정보 + 인터뷰 응답)
  - 리서처 조사 결과 (회사 규모, 기술스택 등)
  - 프로젝트 리서처 산출물 (개인 프로젝트 정리 문서)
  - 에피소드에서 드러나는 패턴 (기술 성향, 강점/약점, 의사결정 스타일)
  - 유저가 제출한 기존 이력서/포트폴리오
- **산출:** 유저 프로파일 문서 — 다른 에이전트에게 공급하여 질문 품질 향상
  - 기술 성향 (FE 중심? 풀스택? 인프라도 만짐?)
  - 커리어 궤적 (성장 방향, 전환 패턴)
  - 강점/약점 맵핑 (JD 대비)
  - 커뮤니케이션 스타일 (답변 패턴에서 추론)
- **실행 타이밍:** 라운드 1 중간, 라운드 2 시작 전, 라운드 3 시작 전 — 시그널이 쌓일 때마다 프로파일 업데이트
- **핵심 가치:** 에이전트들이 유저를 처음 보는 게 아니라 "이미 파악한 상태"에서 질문할 수 있게 만듦

### 시니어 개발자 (senior-dev.md)

- **관점:** 기술 깊이, 구현 디테일, 문제 해결 과정
- **유저 직군에 따라 FE/BE/AI 톤 조절** — 단일 에이전트가 커버
- **질문 예시:** "조사해보니 ㅁㅁ서비스가 React 17에서 18로 마이그레이션했던데, 1) Concurrent Mode 도입 2) Suspense 적용 3) 너는 이 작업 안 했음 — 어디까지 관여했어?"

### CTO (cto.md)

- **관점:** 아키텍처 의사결정, 비즈니스 임팩트, 기술 전략
- **스케일/임팩트 수치 집요하게 추적**
- **질문 예시:** "ㅁㅁ 플랫폼이 MSA 구조인데, 1) 서비스 분리 기준을 네가 설계 2) 기존 구조에서 특정 서비스를 분리하는 작업 담당 3) 모놀리스 유지하면서 모듈만 분리 — 어떤 거였어?"

### 채용담당자 (recruiter.md)

- **관점:** JD 매칭, 시장 경쟁력, 이력서 표현 최적화
- **핵심 역할:** 갭 분석 + 팩폭
- **나이/연차 대비 현실적 피드백** — "솔직히 그 나이에 이 포지션은..."
- **질문 예시:** "토스 시니어 FE JD에 '디자인시스템 경험 필수'인데 관련 에피소드가 없음. 1) 경험 있는데 아직 안 말한 거 2) 진짜 없음 — 어느 쪽?"
- **"진짜 없음"이면:** "이 레벨은 보통 Storybook 기반 컴포넌트 라이브러리 운영 경험이 있음. 갭으로 기록할게."

### 인사담당자 (hr.md)

- **관점:** 소프트스킬, 리더십, 협업, 갈등 해결
- **시니어 포지션일수록 비중 높아짐**
- **질문 예시:** "조사해보니 ㅁㅁ팀이 6명 규모였는데, 1) 주니어 온보딩/멘토링 담당 2) 코드리뷰 문화 주도 3) 기술 의사결정 문서화 리드 — 한 거 있어?"

### 커피챗봇 (coffee-chat/*.md)

- **페르소나 풀에서 랜덤 선택** — 세션마다 다른 사람이 나옴
- **예시 페르소나:** 실리콘밸리 10년차, 스타트업 3회 창업자, 오픈소스 메인테이너, 대기업→스타트업 전환자, 프리랜서 개발자 등
- **톤:** 다른 에이전트보다 캐주얼. 단, 열린 질문 금지 규칙은 동일.
- **역할:** 라운드 3에서 다른 각도로 놓친 에피소드 발굴
- **질문 예시 (스타트업 창업자):** "나도 ㅁㅁ 비슷한 규모에서 일했는데, 그 사이즈면 인프라도 직접 만졌을 거 아냐. 1) CI/CD 파이프라인 구축 2) 모니터링/알림 세팅 3) 배포 자동화 — 뭐 했어?"

## 라운드 구성

하이브리드 구조: 큰 틀은 라운드제, 라운드 내에서는 자유 토론.

### 라운드 0: 세팅

오케스트레이터 단독. 에이전트 안 나옴. 빠르게 끝냄.

1. "이어하기 / 새로 시작" 제시 (기존 resume-source.json이 있으면)
2. 기본 정보 수집 — 빈 칸만 하나씩:
   - 이름 (메모리에 있으면 자동 채움 + 확인)
   - 나이
   - 경력 연수
   - 다녔던 회사 이름들 (한꺼번에 받음)
   - 타겟 회사/포지션
3. 기존 이력서/포트폴리오 수집 (파일, URL, 텍스트 — 선택)
   - 있으면: 파싱 → 충분한 항목은 소스 반영, 부족한 항목은 질문 큐
   - 없으면: 바로 라운드 1
4. 리서처 병렬 실행:
   - 회사당 1인스턴스 (유저 경력 회사들)
   - 타겟 회사/JD 1인스턴스
5. `resume-source.json` 초기 생성 (meta + profile + companies 뼈대)

### 라운드 1: 경력 발굴

- **주도:** 시니어 개발자 + CTO
- **보조:** 리서처 (조사 결과 공급)
- 유저 경력을 회사별로 순회하면서 에피소드 발굴
- 리서처 팩트 기반 구체적 질문 + 선택지 필수
- 기존 소스에서 부족한 항목 집중
- 에피소드 수집할 때마다 resume-source.json에 누적 저장
- **전환 기준:** 오케스트레이터가 "더 뽑을 에피소드가 없다" 판단 또는 유저가 "다음"

### 라운드 2: 임팩트 발굴 + 갭 분석

- **주도:** 채용담당자 + 인사담당자
- **보조:** CTO (기술 기준선 보강)
- 라운드 1에서 모은 에피소드를 JD 기준으로 평가
- 부족하면 팩폭 + 시장 기준선 제시
- 소프트스킬/리더십 에피소드 추가 발굴
- gap_analysis 섹션 추가 저장
- **전환 기준:** JD 주요 요구사항 전부 커버 또는 갭 확인 완료

### 라운드 3: 마무리 + 산출

- **주도:** 커피챗봇 (랜덤 페르소나)
- 편한 톤으로 놓친 에피소드 마지막 체크
- 오케스트레이터가 전체 취합:
  - `resume-source.json` 최종 저장
  - `resume-draft.md` 생성 (갭 피드백 섹션 포함)

### resume-draft.md

JD 맞춤 이력서 초안. 갭 분석 결과를 유저 피드백 섹션으로 포함.

