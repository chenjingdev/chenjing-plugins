# story-spec prototype 세션 retrospective

> 세션 일자: 2026-05-01
> 진행자(인터뷰어): Claude (story-spec skill 실행 중)
> 응답자: chenjing
> 주제: chenjing-plugins 안에 *"흐릿한 아이디어를 구체화해주는 Claude Code 플러그인"* 의 spec을 잡으려 인터뷰하던 중, 응답자가 본인이 만들고자 하는 도구의 prototype(= 지금 실행 중인 story-spec)을 직접 사용하면서 그 prototype의 결함을 발견. 인터뷰는 L2 5/9 시점에서 중단.

## 메타 동형성

- 인터뷰 도구(`story-spec`) ≈ 응답자가 만들려는 플러그인.
- 즉 이 세션은 응답자의 가설 — *"스펙코드 수준으로 구체화되면 하네스 없이 끝까지 자동 구현 가능"* — 의 첫 실증 시도이자, prototype의 사용성 평가이기도 했음.
- 인터뷰어(Claude)가 이 동형성을 자발적으로 짚지 못함. 응답자의 *"지금 너가 실행중인 플러그인이 나랑 인터뷰중인거는 눈치챘어?"* 한마디에야 인지.

## 채워진 산출물 (인터뷰 중단 시점)

- L1 Pixar Pitch 7문장 — **완성** (`docs/spec/` 하위 별도 정리 예정)
- L2 Story Circle 8단계 — **3/8** (You · Need · Go 완성, Search·Find·Take·Return·Change 미정)

## 발견된 문제

### 1. 첫 안내 톤이 너무 문학적

- **증상:** 방법론 옵션 설명이 비유·은유 일색. ("도구·생산성 앱의 *해고 결심* 순간을 또렷이 끌어냅니다" 등)
- **인용:** *"너무 문학적인 표현 말고 알아듣게 말해봐"*
- **가설:** 평이한 한 줄을 default로 두고, 비유는 사용자가 더 깊이 가고 싶다고 신호할 때만 노출. 옵션 카드의 *서술형 한 줄*을 더 짧고 행동 중심으로 (예: "기존 도구를 왜 그만뒀는지를 캐묻는 인터뷰").

### 2. 사용자 단어를 AI가 슬쩍 확장 (Clean Language 위반)

- **증상:** 응답자가 *"발굴 스탭"* 이라 한 걸 다음 메시지에서 AI가 "스탭을 거치며", "스탭이 끝나면" 등으로 확장. "스탭"이 단계 하나인지 전체 묶음인지 미합의 상태에서 후속 문장에 그대로 사용.
- **인용:** *"스탭이라는게 모든 과정을 말하는거야?"*
- **가설:** 사용자가 새로 발화한 단어는 *그 단어로 후속 질문/문장을 짓기 전에 정의를 명시적으로 합의*. 합의 없이 그 단어를 재사용 금지. SKILL.md의 Clean Language 룰이 "다른 단어로 바꾸지 마라"에는 강하지만 "그 단어를 모호한 채로 재사용하지 마라"에는 약함.

### 3. 응답자가 정의를 요청했을 때 AI가 사용자에게 떠넘김

- **증상:** *"스탭이 어떤 단위?"* 명료화 질문에 AI가 *"처음 쓰신 분이 정의하는 게 맞아요"* 로 되돌림.
- **인용:** *"그건 너가 알려줘야지"*
- **가설:** Clean Language의 과적용. 응답자가 자기 단어의 의미를 모를 때는 **AI가 가능한 해석 후보를 제시하고 응답자가 고른다**. 정의 자체를 응답자에게 강요하지 않음.

### 4. 맥락상 자명한 모호함을 굳이 분리해 다시 물음

- **증상:** Need 단계 답변 *"나는 이 앱을 완성할 수 있는 사람"* 에서 *"이 앱"* 의 의미가 직전 5·6번 문장 흐름상 자명한데도 AI가 "이 플러그인 자체인가요, 머릿속 아이디어/앱인가요?"로 분리해 다시 물음.
- **인용:** *"지금 질문이 이상하다고 생각 안해?"*
- **가설:** 모호한 단어를 만났을 때 *(a) 직전 N개 문장의 흐름에서 의미가 추론되는지* 먼저 자체 점검하고, (b) 추론 가능하면 그 추론을 짧게 굳혀 보여주는 식으로 확인. 분리해 다시 묻는 건 흐름 추론으로 안 풀릴 때만.

### 5. 인터뷰 도구와 만들고자 하는 도구의 메타 동형성 미탐지

- **증상:** 응답자가 *"안개처럼 흐릿한 아이디어를 구체화해주는 Claude Code 플러그인"* 을 만들고 있고, 그 인터뷰 도구가 같은 결의 도구인데 AI가 자발적으로 짚지 못함.
- **인용:** *"지금 너가 실행중인 플러그인이 나랑 인터뷰중인거는 눈치챘어?"*
- **가설:** SKILL.md에 *"이 skill 자체와 동형인 도구를 사용자가 만들고 있다면 그 메타 동형성을 명시적으로 짚는다"* 는 룰 부재. 카테고리 제안 단계나 L1 종료 시점에 "이 도구가 우리 인터뷰 도구와 결이 비슷한데, 이 세션 자체를 prototype 사용 경험으로 보겠습니다" 식의 메타 인지 hook 필요.

### 6. 아직 만들지 않은 도구의 시행착오를 상상으로 답하게 만듦

- **증상:** Search 단계에서 *"그가 첫 스탭에 들어가서 처음 몇 번은 어떤 시행착오를 겪을 거 같아요?"* — prototype이 없는 상태에서 응답자가 *상상*으로 미래 사용자의 시행착오를 답해야 함. 이게 인터뷰 중단의 직접 트리거.
- **가설:** Inquiry/도구 카테고리에서 prototype 미존재 시점의 Search 단계는:
  - *유사 자매 도구(GSD·Spec Kit·Superpowers 등) 사용 경험에서 시행착오 추출* 로 좁히거나
  - **이 세션 자체를 prototype 사용 경험으로 받기** (지금 응답자가 한 게 그것).
- 후자가 곧 retrospective 메커니즘이고, 이게 SKILL의 정식 분기로 들어가면 같은 결의 막힘이 다음 세션에서 자동 해결됨.

### 7. 막힘 시점의 후보 제시 타이밍이 한 박자 늦음

- **증상:** Pixar Pitch 5번에서 응답자가 *"뭘 넣으면 좋을까?"* 로 막힘. AI가 두 후보(A/B)를 제시하긴 했지만, 막힘 신호가 처음 들어왔을 때 한 박자 늦게 반응.
- **가설:** 응답자의 막힘 신호 (*"음"*, *"흠"*, *"뭘 넣으면 좋을까"*, *"잘 모르겠어"*) 가 들어오면 즉시 — *직전까지 응답자가 한 발화에서 단서를 끌어와* 후보 2-3개 + 직접 입력 옵션 제시. 후보는 prescription이 아니라 "당신이 이미 한 말의 재조합"임을 명시.

### 8. 압축/풍성도 룰의 모호함

- **증상:** MICE Quotient 결과 *"Inquiry 축이니 You·Go는 가볍게, Need·Search·Find는 깊게"* 로 진행했지만, "가볍게"·"깊게"의 기준이 AI 임의 판단.
- **가설:** MICE 4축별로 Story Circle 8단계 깊이 매트릭스를 수치화 (예: Inquiry 축 → You: 1문장 / Need: 3-5턴 인터뷰 / Find: 5-7턴 / Take: 1문장).

### 9. MICE Quotient 첫 질문이 "어려움 고르라는 거야?"로 오해받음

- **증상:** *"이 사용자 여정의 주된 갈등은…"* 질문에 응답자가 *"그러니까 내가 만든 앱을 이용하면서 겪을 어려움을 고르라는거야?"* 로 의미를 잘못 잡음.
- **가설:** "갈등" → "이 앱을 쓰는 동안 사용자가 *주로 어떤 종류의 변화를 겪는가*"로 평이화. 학술 용어를 응답자에게 노출하지 않는다는 안티패턴 룰이 mice-quotient.md에 이미 있음에도 첫 질문에서 어겨짐.

| 2 | "사용자 단어 정의 미합의 상태에서 그 단어를 후속 문장에 재사용 금지" |

| 8 | "MICE 축별 8단계 깊이 매트릭스 수치화" |
| 9 | "방법론별 첫 질문은 학술 용어 노출 금지 — 평이화 검증을 SKILL 내부에 강제" |

L2 나머지(Search·Find·Take·Return·Change)는 위 문제들이 정리된 *다음 버전의 prototype*에서 다시 채우는 게 의미 있다.
