# scenario schema (L4)

## 무엇인가

`scenario.md`는 **L4 시나리오 카드**의 backbone이다. 시나리오는 한 챕터 안에서 사용자가 *무엇을 하려 하고, 왜, 어떤 조건에서 출발해, 어떤 행동으로, 어떤 장애물을 넘고, 어떻게 성공을 안다*는 한 묶음의 이야기다. Carroll의 Scenario-Based Design을 6필드로 정형화한 것. 사용자 프로젝트에서는 `docs/spec/chapters/<chapter>/scenarios/<id>-<slug>/card.md`로 작성된다.

## 필드 정의

6개 필드. 한 필드당 1-3문장 또는 짧은 목록.

### 1. 목표
- **역할:** 사용자가 이 시나리오에서 *무엇을 달성하려 하는가*.
- **작성 가이드:** 기능명이 아니라 사용자 시점의 한 문장. *"___을 한다"* 형태. 동기(왜)는 다음 필드 몫.
- **좋은 예:** *"머릿속에 떠오른 한 가지 할 일을 5초 안에 화면에 옮겨둔다."*
- **나쁜 예:** *"todo 추가 기능을 사용한다."* (시스템 시점·기능명)

### 2. 동기
- **역할:** 사용자가 *왜* 그 목표를 원하는가. 표면 동기에서 정체성 가치까지.
- **작성 가이드:** Laddering 사고법. *"왜 그게 중요해요?"*를 3-5번 던져 표면 → 기능적 → 정서적 → 정체성 가치까지 내려간다. 1-3문장으로 그 사슬을 압축.
- **좋은 예:** *"잊으면 거래처 신뢰가 깨진다(기능). 그녀는 거래에서 '말한 건 지키는 사람'이라는 정체성을 잃고 싶지 않다(정체성)."*
- **나쁜 예:** *"편리하니까."* (한 단어, 정체성까지 안 내려감)

### 3. 입력 (전제 조건 + 사용자가 가지고 들어오는 것)
- **역할:** 시나리오가 시작되는 순간 *이미 있어야 하는* 상태와 *사용자가 손에 든* 것.
- **작성 가이드:** 시스템 전제(예: 로그인됨)와 사용자 자원(예: 머릿속에 떠오른 한 문장)을 둘 다. 가정이 아니라 관찰 가능한 형태로.
- **좋은 예:** *"전제: DayLine 앱이 메인 화면 상태로 열려 있고 사용자가 로그인됨. 사용자 자원: 방금 통화에서 들은 '내일 견적서 보내기'라는 한 문장이 단기 기억에 있다."*
- **나쁜 예:** *"사용자가 앱을 사용할 준비가 되어 있다."* (관찰 불가, 막연함)

### 4. 행동 (시간 순)
- **역할:** 사용자가 수행하는 일련의 동작. 시간 순서로.
- **작성 가이드:** 3-7개 step. 각 step은 *사용자 행동* 하나(시스템 반응은 5번 필드의 형제 격인 L5 scene에서 다룬다, 여기선 사용자 측만). 클릭·타이핑·말 등 raw 행동.
- **좋은 예:**
  1. 메인 화면 상단의 한 줄 입력창을 탭한다.
  2. *"내일 견적서 보내기"*를 타이핑한다.
  3. Enter를 누른다.
- **나쁜 예:** *"사용자가 todo를 추가한다."* (한 동작으로 압축, 시간 순 분해 없음)

### 5. 장애물 (실패 가능 지점)
- **역할:** 이 시나리오가 *실패하거나 마찰을 겪을 수 있는* 지점들.
- **작성 가이드:** 사용자 측 장애(예: 한 문장이 너무 길어짐)와 시스템 측 장애(예: 입력창이 키보드에 가려짐)를 둘 다. premortem 사고법: *"1년 뒤 이 시나리오가 망했다면 무엇 때문이었을까?"*
- **좋은 예:**
  - 사용자가 한 문장 안에 두 가지 일을 섞어 넣는다 (*"견적서 보내고 카톡 답장"*).
  - 입력 중 다른 알림이 와서 입력 흐름이 끊긴다.
  - Enter가 줄바꿈으로 동작하는지 저장으로 동작하는지 헷갈린다.
- **나쁜 예:** *"버그가 날 수 있다."* (구체적이지 않음)

### 6. 성공 조건 (관찰 가능한 형태)
- **역할:** 이 시나리오가 *성공했음을 어떻게 아는가*. 사용자·시스템 양쪽에서 관찰 가능한 신호.
- **작성 가이드:** 감정·만족이 아니라 관찰 가능 행동/상태. *"___이 보인다"*, *"___이 일어난다"* 형태. 1-3개.
- **좋은 예:**
  - 메인 화면 목록 최상단에 *"내일 견적서 보내기"* 항목이 보인다.
  - 입력창이 비워진 상태로 다시 활성화되어 있다 (연속 입력 가능).
  - 추가 시점부터 5초 안에 사용자가 시야를 다른 앱으로 옮긴다 (캡처 후 이탈 자유도).
- **나쁜 예:** *"사용자가 만족한다."* (관찰 불가)

## 출력 markdown 템플릿

`docs/spec/chapters/<chapter>/scenarios/<id>-<slug>/card.md`에 다음 구조.

```markdown
# Scenario: <한 줄 제목>

> id: <id>
> chapter: <chapter id>
> depth: L4 | L5 | L6
> status: draft | in_progress | approved | implemented
> methodology_l4: <사용한 방법론>

## 목표
<1문장>

## 동기
<1-3문장, 표면→정체성 사슬>

## 입력
- 전제: ...
- 사용자 자원: ...

## 행동 (시간 순)
1. ...
2. ...
3. ...

## 장애물
- ...
- ...

## 성공 조건
- ...
- ...
```

## 가상 todo 앱 "DayLine" 예시 (좋은 예시 1세트)

```markdown
# Scenario: 한 줄로 할 일 추가하기

> id: 01-quick-add
> chapter: 02-first-capture
> depth: L5
> status: approved
> methodology_l4: laddering

## 목표
머릿속에 떠오른 한 가지 할 일을 5초 안에 화면에 옮겨둔다.

## 동기
잊으면 거래처 신뢰가 깨진다. 민지는 거래에서 *"말한 건 지키는 사람"*이라는 정체성을 잃고 싶지 않다 — 도구는 기억의 외주지만, 정체성의 보호막이기도 하다.

## 입력
- 전제: DayLine 앱이 메인 화면 상태로 열려 있고 사용자가 로그인됨.
- 사용자 자원: 방금 통화에서 들은 *"내일 견적서 보내기"*라는 한 문장이 단기 기억에 있다.

## 행동 (시간 순)
1. 메인 화면 상단의 한 줄 입력창을 탭한다.
2. *"내일 견적서 보내기"*를 타이핑한다.
3. Enter를 누른다.

## 장애물
- 사용자가 한 문장 안에 두 가지 일을 섞어 넣는다 (*"견적서 보내고 카톡 답장"*).
- 입력 중 다른 알림이 와서 입력 흐름이 끊긴다.
- Enter가 줄바꿈으로 동작하는지 저장으로 동작하는지 헷갈린다.

## 성공 조건
- 메인 화면 목록 최상단에 *"내일 견적서 보내기"* 항목이 보인다.
- 입력창이 비워진 상태로 다시 활성화되어 있다.
- 추가 시점부터 5초 안에 사용자가 시야를 다른 앱으로 옮긴다.
```

### 나쁜 예시 (대조)
```markdown
# Scenario: todo 기능

## 목표
todo 추가 기능을 사용한다.

## 동기
편리하니까.

## 입력
사용자가 앱을 사용할 준비가 되어 있다.

## 행동
사용자가 todo를 추가한다.

## 장애물
버그가 날 수 있다.

## 성공 조건
사용자가 만족한다.
```

문제점: 모든 필드가 시스템 시점·압축·관찰 불가. backbone이 깨진다.

## AI 행동 룰 (이 schema를 인터뷰 backbone으로 쓸 때)

L4 시나리오 카드를 채울 때 너는 6필드를 *목표 → 동기 → 입력 → 행동 → 장애물 → 성공 조건* 순서로 묻는다. 한 필드당 한 메시지 한 질문 룰. **동기 필드에서는 manifest의 `methodology_l4` (default Laddering)에 따라 *"왜 그게 중요해요?"*를 3-5번 반복해 표면→정체성 사슬을 만든다.** **장애물 필드에서는 Premortem (*"1년 뒤 이 시나리오가 망했다면 왜?"*)을 권한다.** **성공 조건 필드에서는 사용자가 감정형 답을 주면 *"그건 어떤 모습으로 보여요?"*로 관찰 가능 형태로 한 단계 내린다 (Pragmatism).** 6필드가 채워지면 사용자에게 요약 → 명시 승인 → `card.md` 작성 → manifest의 해당 시나리오에 `depth: L4`, `status: in_progress` 또는 `approved`, `methodology_l4: <선택된 방법론>` 갱신.
