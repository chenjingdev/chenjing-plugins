# scene schema (L5)

## 무엇인가

`scene.md`는 **L5 장면 분해**의 backbone이다. 한 시나리오(L4)는 한 개 이상의 장면으로 펼쳐진다. 장면은 *어떤 화면에서, 어떤 상태로 시작해, 사용자가 어떤 raw input을 주고, 시스템이 어떤 반응을 돌려주는지*를 한 호흡 단위로 적은 것. 시나리오 카드가 *시간 순 행동*을 적었다면 장면은 그 행동 1개를 *입력·반응의 표면*까지 펼친다. 사용자 프로젝트에서는 `docs/spec/chapters/<chapter>/scenarios/<scenario>/scenes/<id>-<slug>.md`로 작성된다.

## 필드 정의

4개 필드.

### 1. 화면
- **역할:** 이 장면이 펼쳐지는 화면(또는 surface)의 이름과 정체.
- **작성 가이드:** 화면 이름 + 한 줄 정체. 사용자가 그 화면을 부르는 이름이 있으면 그대로(Clean Language).
- **좋은 예:** *"메인 화면 — 오늘 할 일이 시간 역순 한 줄씩 쌓이는 한 화면. 상단에 한 줄 입력창."*
- **나쁜 예:** *"홈 화면."* (정체 묘사 없음)

### 2. 상태 (장면 직전 시스템 상태)
- **역할:** 사용자 input이 들어오기 *직전*의 시스템 상태. 무엇이 켜져/꺼져/로딩/포커스 상태인지.
- **작성 가이드:** 사용자가 보는 것(visual) + 시스템이 가진 것(data) 둘 다. 짧은 목록.
- **좋은 예:**
  - 메인 화면이 표시됨, 목록에 어제까지의 항목 7개.
  - 한 줄 입력창은 placeholder *"한 줄로 적어요"* 표시, 포커스 없음.
  - 키보드는 닫힘.
- **나쁜 예:** *"앱이 열려 있다."* (직전 상태를 관찰 가능 형태로 적지 않음)

### 3. 사용자 행동 (raw input)
- **역할:** 사용자가 이 장면에서 주는 *물리적/음성적 입력*. 클릭·탭·타이핑·말·시선 등.
- **작성 가이드:** *의도*가 아니라 *관찰 가능한 동작 한 개*. *"입력창을 탭"*은 OK, *"todo를 추가하려 함"*은 NO.
- **좋은 예:** *"입력창을 한 번 탭한다."*
- **나쁜 예:** *"todo를 추가하려고 한다."* (의도, 관찰 불가)

### 4. 시스템 반응 (시각·청각·데이터 변화)
- **역할:** 사용자 input에 대한 시스템의 반응 — 시각·청각·햅틱 변화 + 데이터 변화 둘 다.
- **작성 가이드:** *visible* 변화와 *invisible* 변화를 분리해서 적는다. visible은 사용자가 보고/듣고/느끼는 것, invisible은 데이터·상태 전이.
- **좋은 예:**
  - visible: 입력창에 포커스가 들어오며 placeholder가 사라짐. 키보드가 화면 하단에서 위로 올라옴.
  - invisible: UI 상태 `idle` → `composing`. 새 todo 객체는 아직 생성되지 않음 (Enter까지 대기).
- **나쁜 예:** *"키보드가 뜬다."* (invisible 누락, 한 줄 압축)

## 시퀀스 다이어그램 (옵션)

여러 장면이 빠르게 연쇄될 때 — 특히 L5에서 한 시나리오가 3개 이상 장면으로 쪼개질 때 — 시퀀스 다이어그램을 텍스트 형식으로 곁들이면 좋다. 의무 아님.

```
사용자          UI                       데이터
  |              |                          |
  | tap 입력창   |                          |
  |------------->| focus, 키보드 up         |
  |              | state: idle→composing    |
  | type "..."   |                          |
  |------------->| 텍스트 표시              |
  |              |                          |
  | press Enter  |                          |
  |------------->|                          |
  |              |---- create todo -------->|
  |              |<--- todo {id, text} ----|
  |              | 목록 최상단에 추가       |
  |              | 입력창 비우고 재활성     |
```

## 출력 markdown 템플릿

`docs/spec/chapters/<chapter>/scenarios/<scenario>/scenes/<id>-<slug>.md`에 다음 구조.

```markdown
# Scene: <한 줄 제목>

> id: <id>
> scenario: <scenario id>
> order: <장면 순서 번호>

## 화면
<이름 — 한 줄 정체>

## 상태 (직전)
- visible: ...
- data: ...

## 사용자 행동
<raw input 한 개>

## 시스템 반응
- visible: ...
- invisible: ...

## (옵션) 시퀀스 다이어그램
```
<텍스트 다이어그램>
```
```

## 가상 todo 앱 "DayLine" 예시 (장면 1개: "할 일 추가" — Enter 시점)

```markdown
# Scene: Enter로 한 줄을 목록에 떨군다

> id: 03-press-enter
> scenario: 01-quick-add
> order: 3

## 화면
메인 화면 — 오늘 할 일이 시간 역순 한 줄씩 쌓이는 한 화면. 상단에 한 줄 입력창.

## 상태 (직전)
- visible: 입력창에 *"내일 견적서 보내기"* 텍스트가 표시되어 있고 커서가 문장 끝에 있음. 키보드 열림.
- data: UI 상태 `composing`. 임시 텍스트는 메모리에 있을 뿐 todo 객체는 아직 없음.

## 사용자 행동
Enter 키를 누른다 (모바일이라면 키보드의 *완료* 또는 *↵*).

## 시스템 반응
- visible: 목록 최상단에 *"내일 견적서 보내기"* 항목이 부드럽게 한 줄 끼어들며 추가됨. 입력창은 비워지고 placeholder *"한 줄로 적어요"*가 다시 표시됨. 입력창 포커스는 유지됨 (연속 입력 가능). 가벼운 햅틱 1회.
- invisible: 새 todo 객체 `{id, text, created_at, status: open}` 생성·저장. UI 상태 `composing` → `idle`. 목록 캐시 무효화 후 재렌더.

## 시퀀스 다이어그램
```
사용자        UI                          데이터
  |            |                             |
  | press Enter|                             |
  |----------->|                             |
  |            |---- create todo ----------->|
  |            |<--- todo {id, text, ts} ----|
  |            | 목록 prepend                |
  |            | 입력창 clear & re-focus     |
  |            | 햅틱 1회                    |
```
```

## AI 행동 룰 (이 schema를 인터뷰 backbone으로 쓸 때)

L5 장면을 채울 때 너는 4필드를 *화면 → 상태(직전) → 사용자 행동 → 시스템 반응* 순서로 묻는다. 한 필드당 한 메시지 한 질문 룰. **사용자 행동 필드에서 사용자가 *의도형* 답(*"todo를 추가한다"*)을 주면 manifest의 `methodology_l5` (default Master/Apprentice)에 따라 *"제가 견습생이라 치고, 화면에서 손이 어디로 먼저 가요? 무엇을 누르나요?"*로 raw input까지 한 단계 내린다.** **시스템 반응 필드에서는 visible과 invisible을 *둘 다* 묻는다 — 사용자가 visible만 답하면 *"그 사이에 데이터 쪽에서는 무엇이 바뀌었을까요?"*로 invisible을 채운다.** 한 시나리오는 보통 2-5개 장면으로 펼쳐지므로 사용자에게 *"이 장면 하나로 끝낼까요, 더 펼칠까요?"*를 묻고 사용자가 그만 펼치라고 할 때까지 새 장면을 추가한다(자기 판단으로 멋대로 더 쪼개거나 합치지 않는다). 한 장면이 채워지면 사용자에게 요약 → 명시 승인 → 해당 scene 파일 작성 → manifest의 시나리오 항목 `depth: L5` 갱신.
