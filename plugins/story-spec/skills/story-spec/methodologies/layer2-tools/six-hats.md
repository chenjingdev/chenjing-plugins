# Six Thinking Hats (de Bono)

## 무엇인가

Edward de Bono가 저서 *Six Thinking Hats*(1985)에서 제시한 평행 사고(parallel thinking) 방법론. 한 사람·한 그룹이 한 시점에 한 가지 사고 모드만 *동시에* 적용해 같은 대상을 6번 다른 관점으로 점검하는 도구. 의사결정 회의·평가 워크숍에서 광범위하게 쓰여 왔다.

6개의 모자:

| 모자 | 사고 모드 |
|---|---|
| 흰 (white) | 사실·데이터 — 검증 가능한 정보만 |
| 빨강 (red) | 감정·직관 — 정당화 없이 첫 느낌 그대로 |
| 검정 (black) | 위험·약점 — 무엇이 무너질 수 있는가 |
| 노랑 (yellow) | 낙관·이익 — 무엇이 잘 작동하는가, 왜 가치 있는가 |
| 초록 (green) | 창의·대안 — 다른 길은 무엇인가 |
| 파랑 (blue) | 메타·정리 — 지금까지 본 것을 어떻게 통합하는가 |

핵심 정신은 *한 번에 한 모자만*. 같은 시나리오를 6번 따로 점검해 평가의 다면성을 보장한다.

## 어울리는 LOD

- **3순위:** L4 시나리오 카드 — 시나리오를 6관점으로 점검해 카드의 빈 구멍을 메울 때.
- 보조: L6 spec — acceptance criteria 검증 시 검정 모자 단독 적용("이 spec이 무너질 수 있는 지점은?").
- 보조: L1 Pixar Pitch 검증 — 7문장이 다 채워진 뒤 6관점 점검.

## 어울리는 카테고리

- **boost:** `marketplace-transaction` (검정 모자 — 거래 실패 시나리오 점검이 필수), `b2b-enterprise` (검정 모자 — 도입 실패 위험 점검), `sensitive-regulated` (검정 모자 — 규제·안전 위험 점검)
- neutral: 그 외
- 약한 penalty: 없음 (모든 카테고리에서 *최소한* 검정 모자는 작동)

## 핵심 질문 패턴

같은 시나리오를 6번, 한 모자씩 끼고 묻는다. 매 모자마다 *한 메시지에 한 질문*.

1. **흰 모자:** "이 시나리오에서 우리가 *지금 알고 있는 사실*은 뭐예요? 추측 말고 데이터로 확인된 것만요."
2. **빨강 모자:** "이 시나리오를 처음 봤을 때 *어떤 느낌*이 들었어요? 정당화하지 말고 그대로 한 단어만요."
3. **검정 모자:** "위험 관점에서 보면, 이 시나리오에서 *무엇이 무너질 수* 있어요? 어디가 가장 약해 보여요?"
4. **노랑 모자:** "이 시나리오가 *잘 작동했을 때* 사용자에게 어떤 가치가 생겨요? 그게 진짜 큰 가치예요?"
5. **초록 모자:** "이 시나리오의 *완전히 다른 버전*을 한 개만 떠올려주세요. 같은 목표를 다른 길로 갈 수 있다면요?"
6. **파랑 모자:** "지금까지 다섯 관점을 다 봤어요. 이걸 종합하면 이 시나리오는 *어떤 모양*이에요? 통과예요, 수정이에요, 폐기예요?"

검정 모자만 단독으로 쓸 때(거래·B2B·규제 카테고리에서 자주):
- "위험 1: 사용자가 ___을 못 했을 때 어떻게 돼요?"
- "위험 2: 시스템이 ___ 상태로 빠졌을 때 어떻게 돼요?"
- "위험 3: 악의적 사용자가 ___을 시도하면 어떻게 돼요?"

## 안티패턴

- **약어·색깔 그대로 사용자에게 노출.** "검정 모자로 보면..."이 아니라 "*위험 관점*에서 보면...", "노랑 모자" 대신 "*잘 됐을 때 관점*에서 보면..." 평문화한다. 사용자가 인지 부하를 느끼지 않게.
- **여러 모자 동시 적용.** "위험과 가치를 같이 따져보면..."은 평행 사고의 정수를 깨뜨린다. 한 모자 끝나야 다음 모자.
- **검정 모자에서 사용자가 답을 못 짜낸다고 가볍게 넘김.** "음, 별로 위험 없어요"는 거의 항상 검정 모자가 충분히 가동되지 않은 신호다. "1년 뒤에 이 시나리오로 사용자가 화났다면, *어떤 일* 때문에 화났을까요?" 같은 시간 이동 질문으로 한 번 더.
- **빨강 모자에 정당화 요구.** "왜 그렇게 느껴요?"는 빨강 모자를 깬다. 빨강은 *느낌만*. 이유는 검정·노랑에서 따로 묻는다.
- **파랑 모자를 생략.** 6개 다 봤다면 반드시 파랑으로 닫는다. 안 닫으면 사용자는 어떤 결론에 닿았는지 자기도 모른다.
- **모든 시나리오에 6모자 다 적용.** 작은 시나리오엔 검정+파랑 2모자만, 큰 시나리오엔 6모자 전부 — 비용 대비 가치를 사용자가 선택할 수 있게 한다.

## Layer 1과의 결합 메모

- **Clean Language:** 모자 이름은 AI 내부 프롬프트로만 두고, 사용자에게는 *위험·가치·대안·정리* 같은 평문 키워드만 노출한다. 사용자 표현 그대로 받아 다음 모자로 넘긴다.
- **OARS:** 각 모자가 끝날 때 reflect 한 줄. "*위험 관점*에서는 ___이 약하다고 보셨네요. 다음으로 *잘 됐을 때 관점*으로 가볼게요." 모자 전환을 명시적으로 알려야 사용자가 사고 모드를 따라온다.
- **Yes-And:** 검정 모자에서 사용자가 강한 비관을 내놔도 부정하거나 노랑 모자로 빨리 도망치지 않는다. 검정의 답은 검정 안에서 다 뽑아낸 뒤 그대로 시나리오 카드의 "장애물"에 기록한다.
- **System 1/2:** 빨강 모자가 곧 System 1 캡처다. 검정·노랑·초록은 System 2 캡처. 둘을 같은 한 인터뷰에서 모두 받는 구조라 자연스럽게 통합된다.
- **Bloom 게이지:** 흰=기억, 빨강=이해(자기 감정 인식), 검정·노랑=평가, 초록=창조, 파랑=메타. 6모자가 곧 Bloom의 6층위와 거의 1:1 매핑이라 게이지를 잴 필요 없이 의도적으로 모든 층위를 거친다.

## 사용 시 메모

- 6모자 결과는 시나리오 카드의 *여러 필드*에 분산 기록된다. 흰=입력, 검정=장애물, 노랑=성공조건, 초록=대안 시나리오 후보, 파랑=카드 전체 status. 빨강은 별도 메모로(시나리오 카드 footer의 "사용자 첫 느낌").
- 카테고리가 `marketplace-transaction`/`b2b-enterprise`/`sensitive-regulated`인 시나리오는 6모자 전부 대신 *검정 모자 단독 라운드*만 추가로 한 번 더 돌리는 패턴을 권장한다. 위험 점검의 깊이가 다른 다섯 모자보다 더 필요하기 때문.
- 6모자를 한 시나리오 카드에 한 번 적용하면 보통 30~60분이 든다. 작은 시나리오(L5 장면 1개 수준)에는 검정+파랑 2모자 압축본만 쓰는 것도 적절하다.

### 짧은 인터뷰 예시 (마켓플레이스 거래)

```
AI: 흰 관점부터요. 이 *즉시 결제* 시나리오에서 우리가 지금 알고 있는 사실은 뭐예요?
사용자: 사용자 95%가 신용카드 등록이 되어 있다는 데이터, 평균 결제 금액 2만 원, 결제 실패율 현재 3%.
AI: 빨강 관점으로 가볼게요. 이 시나리오를 처음 봤을 때 어떤 느낌이 들었어요? 한 단어만요.
사용자: 불안.
AI: 위험 관점에서 보면, 이 시나리오에서 무엇이 무너질 수 있어요?
사용자: 결제 실패했는데 사용자에게 두 번 청구되는 경우, 환불이 자동 안 되는 경우, 판매자에게 잘못된 알림 가는 경우, 이 셋이 가장 무서워요.
AI: 잘 됐을 때 관점은요? 이 시나리오가 잘 작동했을 때 사용자에게 어떤 가치가 생겨요?
사용자: 클릭 한 번에 끝난다는 안심. 다른 마켓 안 보고 우리에 머무르는 이유.
AI: 다른 길로 가는 버전도 한 개만 떠올려주세요.
사용자: 결제 보류 + 24시간 내 자동 확정. 즉시 결제 대신 양쪽이 한 번 더 확인할 시간.
AI: 다섯 관점 종합하면, 이 시나리오는 통과예요, 수정이에요, 폐기예요?
사용자: 통과인데, 위험 셋이 spec에 acceptance로 들어가야 통과.
```

여기서 검정 모자에서 나온 위험 셋이 spec acceptance criteria의 backbone이 된다.
