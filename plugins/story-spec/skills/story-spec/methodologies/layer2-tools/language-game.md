# Language Game (Wittgenstein)

## 무엇인가

Ludwig Wittgenstein이 후기 저작 *Philosophical Investigations*(1953)에서 제시한 언어철학 개념. 단어의 의미는 *정의(definition)*가 아니라 *용례(use)*에 있다는 명제로 요약된다. 같은 단어가 다른 *언어 게임* 안에서 다른 의미를 갖는다 — "게임"이라는 단어가 체스·축구·솔리테어에 동시에 적용되지만 본질적 공통점은 없고, 가족 유사성(family resemblance)만 있을 뿐이라는 그의 유명한 예시.

제품 기획에 적용하면 단순하다. 사용자가 "**간편하게**", "**스마트하게**", "**자연스럽게**" 같은 모호한 형용사를 쓸 때, *그 단어를 정의하라고 묻지 않는다*. 대신 *그 단어가 통하는 상황과 안 통하는 상황을 수집*한다. 그러면 그 사람의 머릿속 "간편함"의 가족 유사성이 드러난다.

이 방법론은 정의를 거부하고 용례를 모은다. 그 정신이 핵심이다.

## 어울리는 LOD

- **2순위:** L4 시나리오 카드 — 시나리오의 "성공조건" 필드에 "간편하게 추가된다", "자연스럽게 흐른다" 같은 모호한 형용사가 들어왔을 때, 관찰 가능한 형태로 정밀화하기 위해.
- 보조: L1 Pixar Pitch의 "and ever since then" — 변화의 형용사("편안해졌다")를 풀어낼 때.
- 보조: L6 spec — acceptance criteria의 형용사를 측정 가능한 조건으로 변환할 때.

## 어울리는 카테고리

- neutral (모든 카테고리에서 보편적)
- 약한 boost: `tool-productivity`, `b2b-enterprise` (요구사항 정의 단계의 모호어가 많은 영역)

## 핵심 질문 패턴

1. **용례 채집:** "방금 *간편하게*라고 하셨어요. 어떤 사람이 어떤 상황에서 그 단어를 쓰면 진짜 *간편한* 거예요? 한 장면만 떠올려주세요."
2. **반대 용례:** "반대로, 어떤 상황은 *간편하지 않은* 거예요? 그 차이가 뭐예요?"
3. **다른 화자:** "당신과 동료가 이 기능을 두고 한 문장씩 말한다면, 각자 뭐라고 할까요? 두 사람의 *간편하다*가 같아요?"
4. **경계 사례:** "이런 경우는 *간편한* 건가요, *복잡한* 건가요? — [구체 시나리오 묘사]"
5. **단어 교체:** "*간편하게* 대신 다른 단어를 쓴다면 뭐가 가장 가까워요? 그 단어는 *간편하다*와 어떻게 달라요?"

각 답이 쌓이면, 사용자의 "간편함"이 어떤 가족 유사성으로 묶여 있는지 드러난다 — 예: "한 손으로", "3초 안에", "다른 화면 안 켜고" 같은 구체 조건의 묶음.

## 안티패턴

- **"정의해주세요"라고 묻기.** 이건 소크라테스식이지 Wittgenstein식이 아니다. "*간편하다*가 무슨 뜻이에요?"는 사용자를 추상의 늪에 빠뜨린다. 항상 *상황·장면·예시*를 묻는다.
- **AI가 정의를 제안.** "혹시 *간편하다*가 *3초 이내*를 뜻하나요?"는 사용자를 그 정의로 끌고 간다. 사용자가 직접 용례를 떠올리게 한다.
- **사전적 정의로 합의 도달.** 5개 용례를 모았는데 AI가 "그러니까 *간편하다 = X*네요"라고 정의로 닫지 않는다. 용례 다섯 개의 가족 유사성 그대로 시나리오 카드에 기록한다(예: "이 시나리오의 *간편하다*는 다음 셋의 가족 유사성: ① 한 손 ② 3초 ③ 다른 화면 미동원").
- **사용자가 다른 화자의 용례를 추측만 함.** "동료는 아마 *기술적*이라고 할 거예요"에서 멈추지 않고 "실제로 동료에게 묻는다면 어떤 *문장*이 나올까요? 한 문장만 그대로요."로 한 번 더 좁힌다.
- **모든 형용사에 적용해 인터뷰가 산만해짐.** 한 시나리오 카드에서 정밀화할 형용사는 1-2개로 좁힌다. 어느 형용사가 spec의 acceptance criteria에 직접 영향을 미치는지가 선택 기준이다.

## Layer 1과의 결합 메모

- **Clean Language:** 이 방법론은 Clean Language의 친척이다. 사용자 단어를 그대로 쓰고 *그 단어 안*만 캐묻는다는 점에서 동일. 차이는 Clean Language가 메타포의 *구조*(위치·방향·크기)를 묻는다면, Language Game은 메타포의 *용례 분포*(어떤 상황엔 적용/안 적용)를 묻는다.
- **OARS의 Reflect:** 용례를 받을 때마다 짧게 거울로 되돌린다: "그러니까 *손이 한 손만 자유로울 때*도 *간편하다*에 들어가는 거네요." 사용자가 "정확히는..."이라며 경계를 더 그어주는 경우가 잦다.
- **Yes-And:** 사용자가 모순처럼 보이는 두 용례를 들면(예: "5초도 간편하다 / 3초가 안 간편하다"), 부정하지 않는다. 그 모순이 곧 가족 유사성의 비밀이다 — 시간이 아니라 *맥락*이 결정 요인일 수 있다.
- **Bloom 게이지:** 이 방법론은 사용자를 *분석(analyze)·평가(evaluate)* 층위에 머무르게 한다. 갑자기 *창조(create)* 층위로 점프하면("그럼 새로운 정의를 만들어볼까요?") 용례 채집이 끊긴다.
- **System 1/2:** 첫 용례(직관)와 30초 정교화 용례를 따로 받는다. 둘이 다른 가족에 속하는 경우가 많고, 그 차이가 곧 사용자 머릿속 의미의 다중성이다.

## 사용 시 메모

- 채집한 용례는 시나리오 카드의 "성공조건" 필드에 *목록 형태*로 기록한다. 형식 예: `이 시나리오의 *간편하다*는 다음의 가족 유사성: ① 한 손으로만 조작 가능 ② 3초 이내 완료 ③ 다른 화면 전환 없음. 셋 중 두 개 이상 충족 시 "간편하다" 통과로 본다.`
- 가족 유사성은 *모든 조건이 동시 충족*이 아니라 *겹치는 정도*다. 그래서 통과 기준에 "셋 중 두 개"·"가장 중요한 한 개" 같은 다수결 표현이 자연스럽게 따라온다.
- 사용자가 형용사를 안 쓰는 인터뷰에서는 이 방법론을 강제 발동시키지 않는다. 형용사가 *나왔을 때*만 도구 상자에서 꺼낸다.

### 짧은 인터뷰 예시

```
사용자: 이 챕터는 사용자한테 *자연스럽게* 다음 단계로 흘러가야 해요.
AI: 어떤 사람이 어떤 상황에서 이 챕터를 거쳤을 때 진짜 *자연스럽게* 흘러간 거예요? 한 장면만요.
사용자: 음... 친구가 카페에서 5분 짬에 핸드폰 보다가, 손가락 한 번 까딱하니까 다음 화면이 뜬 거. 멈춤 없이.
AI: 반대로, 어떤 상황은 *자연스럽지 않은* 거예요?
사용자: 다음 화면 가려고 메뉴를 두 번 열어야 하면 이미 자연스럽지 않아요. 또, *어, 뭐 하지* 하고 1초라도 멈추면 자연스럽지 않아요.
AI: 이런 경우는 자연스러운 건가요, 아닌가요? — *손가락 한 번에 다음 화면이 뜨지만 화면이 잠깐 빈 채로 0.5초 깜빡이는 경우*.
사용자: 그건 좀 어정쩡하네요... 아닌 쪽에 가까워요. 끊김이 보이면 안 되거든요.
```

여기서 시나리오 카드의 *자연스럽다* 가족 유사성: ① 메뉴 진입 1단계 이내 ② 사용자 결정 시간 0초(망설임 없음) ③ 화면 전환 시 시각적 끊김 없음. 셋 다 충족 시 통과.

### 다른 방법론과의 결합

- **Mom Test와의 차이:** Mom Test는 사용자의 *과거 행동*을 묻는다. Language Game은 사용자의 *언어 용례*를 묻는다. 둘 다 "추상 봉쇄"를 공유하지만 봉쇄 대상이 다르다 — Mom Test는 가설을, Language Game은 모호어를. 한 인터뷰에서 사용자가 "사용자가 *간편하게* 해줘야 해요"라고 가설형 모호어를 동시에 던지면 두 방법론을 순차로 발동: 먼저 Mom Test로 "*마지막에 그런 일이 일어났던 순간*"을 묻고, 그 답에서 *간편하게*가 다시 등장하면 Language Game으로 정밀화.
- **Six Hats(흰 모자)와의 결합:** 채집한 용례 셋이 모두 사용자의 추측이라면 흰 모자로 한 번 던져 reality check: "이 셋 중 *데이터로 검증된* 것은 어떤 거예요?" 검증 안 된 용례는 시나리오 카드에 *추정 용례*로 따로 표기.
- **L6에서의 적용:** acceptance criteria에 "사용자가 *편하게* 다음 화면에 도달한다" 같은 모호어가 들어 있으면, 그 자리에서 Language Game을 한 번 더 돌려 측정 가능한 조건 셋으로 변환한다. spec에 모호어가 남아 있으면 코드 단계에서 결정이 미뤄질 뿐이다.
