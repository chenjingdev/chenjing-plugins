# JTBD Switch Interview (도입 전환 인터뷰)

## 무엇인가

Bob Moesta와 Chris Spiek이 Re-Wired Group에서 Clayton Christensen의 Jobs-to-be-Done 이론을 인터뷰 프로토콜로 구체화한 기법(2000년대 후반~2010년대 초). 사용자가 *기존 방식을 해고하고 새 방식을 채용한 순간*의 실제 타임라인을 분(分) 단위로 재구성해, 도입 결정의 진짜 힘(push·pull·anxiety·habit)을 분리해 본다.

story-spec에서는 L1의 "시작" — Pixar Pitch의 "Once upon a time"·"Every day"·"One day"를 끌어낼 때 사용한다. **사용자가 처음 '바꿔야겠다'고 생각한 순간 직전의 trigger event를 정확히 짚는 것**이 핵심.

## 어울리는 LOD

- **L1 시작과 끝 — 2순위** (1순위 Miracle Question이 "끝"에 강하다면 JTBD Switch는 "시작"에 강하다)
- L3 챕터의 "Need" 단계 시드 발굴
- L6 acceptance criteria 중 "도입 트리거가 발생했을 때의 시스템 행동" 정의 보조

## 어울리는 카테고리 (boost)

- **tool-productivity** — Notion·Linear·Todoist류는 사용자가 *왜 기존 도구에서 갈아탔는지*가 가장 중요한 도입 결정
- **b2b-enterprise** — B2B 도입은 항상 기존 솔루션·엑셀·이메일·다른 SaaS에서의 *전환*. 그 전환의 결정 회의·결정 자료를 짚어낼 수 있다
- **marketplace-transaction** — 사용자가 기존 채널(전화·오프라인·다른 앱)을 떠나 이 앱으로 거래를 옮기는 동기 분석에 강함

penalty: content-media (콘텐츠는 "전환"보다 "발견"이 본질이라 Master/Apprentice·Projective가 더 적합)

## Pixar Pitch와의 동형 구조

JTBD Switch의 4-force 타임라인은 Pixar Pitch의 도입부와 그대로 매핑된다:

| JTBD 타임라인 | Pixar Pitch 문장 |
|---|---|
| 첫 생각(First thought) — "이거 바꿔야겠다" | One day... |
| 평소 상태 — push of the situation | Every day... |
| 평소 상태 이전 — 만족 환경 | Once upon a time... |
| 결정 순간(Decision) — 대안 비교·선택 | Because of that... |
| 도입 직후 — 첫 사용 | Because of that... |

이 동형성을 이용해 인터뷰 한 번으로 Pixar Pitch 1~5번 문장이 동시에 채워진다.

## 핵심 질문 패턴

1. "*처음으로* '이거 바꿔야겠다'고 생각한 순간이 언제였어요? 날짜·시간·장소 기억나는 만큼 떠올려보세요. 그 직전에 무슨 일이 있었어요?"
2. "그 순간이 오기 전, 평소에는 이 일을 어떻게 하고 있었어요? 평소의 하루를 분 단위로 보여주세요."
3. "그 '바꿔야겠다' 순간에, 어떤 *대안*들을 후보로 떠올렸어요? 머릿속에 떠오른 순서대로요. 각 대안을 왜 고려했고, 왜 결국 선택하지 않았어요?"
4. "그 대안들 중에서 무엇 때문에 결국 *이 방식*을 택했어요? 결정적인 한 가지를 꼽는다면요?"
5. "현재 방식을 *해고*하려면 무엇이 더 일어나야 해요? 지금 이 방식을 계속 쓰게 만드는 *불안*이나 *습관*은 무엇인가요?"

## 안티패턴

- **현재 시점의 일반론 묻기** — "보통 이 도구를 어떻게 쓰세요?" / "어떤 기능이 좋아요?" 같은 질문은 JTBD가 아니다. 반드시 *과거 한 순간의 구체적 사건*을 잡아야 한다.
- **trigger event를 "필요"로 일반화** — "필요해서요"라는 답을 그대로 받기. "필요해진 *그 순간*에 무엇이 일어났는지"를 끝까지 캐묻는다.
- **대안 비교를 생략** — 사용자가 곧장 "이 앱이 좋아서 골랐어요"라고 하면, 반드시 "그 결정 직전에 다른 어떤 후보가 있었어요?"로 *경쟁 집합*을 복원한다. 경쟁 집합이 없는 도입 인터뷰는 JTBD가 아니다.
- **불안·습관 force를 묻지 않음** — 도입을 *촉진*한 force만 묻고 *방해*한 force(현재 방식의 익숙함, 새 방식의 학습 비용 등)를 빼먹기. 4-force 모두 잡아야 진짜 결정 풍경이 나온다.
- **사용자 가설("아마 X 때문일 거예요")을 사실로 기록** — "아마"·"~인 것 같아요" 답은 추가 캐물음으로 사건 단위까지 내려야 한다.

## Layer 1 메타 자세와의 결합 메모

- **Clean Language와의 결합** — JTBD는 *시간 축*을 캐묻는 기법이라 Clean Language의 "그 [사용자 단어] *직전*에는 무엇이?"·"*직후*에는 무엇이?" 패턴과 완벽히 합쳐진다. 사용자가 "짜증이 폭발했어요"라고 하면 끝까지 "짜증"·"폭발"을 그 단어로 두고 ("화"·"분노"로 바꾸지 않고) 직전·직후·옆을 캐묻는다.
- **OARS의 Summarize** — 4-force 타임라인이 어느 정도 채워지면 한 문장으로 요약해 되돌려준다 ("그러니까, *수요일 오후 클라이언트 미팅 직후*에 '엑셀로는 더 못 한다' 싶었고, Notion·Airtable·Coda를 떠올렸는데 *팀이 이미 Notion을 쓰고 있어서* 결정했다 — 맞나요?"). 사용자가 자기 정정할 기회.
- **Mom Test와 보완** — JTBD가 "왜 바꿨나"를 묻고, Mom Test가 "그 답이 망상이 아닌가"를 검증한다. 두 방법론을 같은 L1 노드에서 *순차*로 쓰면 강력하다(JTBD로 끌어내고, Mom Test 룰로 가정형 답을 차단).
- **System 1/2 이중 캡처** — "그 순간에 *처음 떠오른* 대안은?" (System 1) "그다음 30초 동안 추가로 떠올린 후보는?" (System 2). 둘 다 기록.
- **Yes-And** — 사용자가 trigger event를 묘사할 때 "그건 좀 사소한데요?" 같은 평가를 절대 하지 않는다. 사소해 보이는 trigger가 진짜 도입 결정의 핵심인 경우가 다수.
