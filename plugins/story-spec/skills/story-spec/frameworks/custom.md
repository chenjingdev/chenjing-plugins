# Custom Framework

## 무엇인가

기본 framework 쌍(Pixar Pitch + Story Circle)이나 대안(Kishōtenketsu·Hero's Journey)이 사용자의 앱·취향·문화에 맞지 않을 때, 사용자는 자기만의 framework를 정의해 쓸 수 있다. 사용자는 `docs/spec/framework.md`에 직접 본문을 작성하고, AI는 그 framework를 backbone으로 인터뷰를 진행한다.

이 옵션은 *드물게* 권장된다. 기본 쌍은 영화·문학에서 검증된 활(arc)을 제공하고, 대안 두 가지로 카테고리별 맞춤까지 커버한다. Custom은 *사용자가 자기 도메인에 더 적합한 활을 이미 가지고 있을 때*만 의미가 있다 — 예: 의료 앱에 SOAP 노트 4단계를 적용, 교육 앱에 Kolb의 경험 학습 4사이클을 적용, 시(詩) 앱에 시조의 초장-중장-종장 3단계를 적용.

## 사용자가 작성하는 것

`docs/spec/framework.md`에 다음 두 가지를 자유 형식으로 작성한다. AI가 이 파일을 읽어 인터뷰 backbone으로 삼는다.

### 최소 요구

1. **L1 단계 정의 — 시작과 끝 한 쌍.**
   - 시작 단계 1개 + 끝 단계 1개 (총 2단계 이상이면 최소 충족)
   - 각 단계의 *핵심 질문* 1-2개 (사용자가 무엇에 답해야 하는가)
   - Pixar Pitch의 1번 *Once upon a time*과 7번 *And ever since then*에 대응

2. **L2 펼침 단계 정의 — N단계 (3 ≤ N ≤ 12 권장).**
   - 시작과 끝 사이를 채우는 중간 단계들
   - 각 단계의 *핵심 질문* 1개 이상
   - Story Circle의 8단계처럼 *순서가 의미를 만드는* 단계들이어야 함

선택 추가 정보 (있으면 인터뷰 품질이 올라간다):
- framework의 출처·역사 1-2문장
- 어울리는 앱 카테고리
- 챕터 시드로의 변환 가이드 (각 L2 단계가 어떤 L3 챕터로 펼쳐지는가)

### 작성 예시: SOAP 기반 의료 앱 framework

```markdown
# 우리 앱 framework: SOAP arc

## 출처
의료 차팅의 표준 SOAP 노트(Subjective·Objective·Assessment·Plan)를
환자 사용자 여정에 적용.

## L1 시작과 끝
1. **Pre-visit** — 환자는 어떤 증상·불안을 가지고 앱에 들어왔나
2. **Post-visit** — 환자는 무엇을 들고 떠나나 (처방·계획·안심)

## L2 펼침 4단계
1. **S — Subjective**: 환자가 자기 말로 증상을 설명한다
2. **O — Objective**: 앱이 측정·기록 가능한 데이터를 수집한다
3. **A — Assessment**: 의료진(또는 알고리즘)이 진단·평가를 내놓는다
4. **P — Plan**: 환자와 의료진이 다음 행동을 합의한다

## 어울리는 카테고리
sensitive-regulated (1순위)
```

이 정도 분량이면 충분하다. 2-4단락이 권장 길이다.

## AI의 행동

사용자가 `docs/spec/framework.md`를 작성하면 AI는 다음과 같이 작동한다.

1. **L1 진입 시** `docs/spec/framework.md`를 읽어 L1 단계 정의를 추출. Pixar Pitch 7문장 대신 사용자가 정의한 시작·끝 단계를 backbone으로 인터뷰.
2. **L2 진입 시** 사용자가 정의한 N단계를 backbone으로 인터뷰. Story Circle 8단계 대신 N단계를 순서대로 묻는다.
3. **L3 진입 시** N단계가 그대로 챕터 후보 N개로 펼쳐진다. 사용자가 정의한 챕터 변환 가이드가 있으면 그대로 따르고, 없으면 *"이 N단계를 N개 챕터로 만들어볼까요? 합치거나 나누고 싶은 단계가 있나요?"* 묻는다.
4. **manifest 기록.** `framework: custom` 으로 적고, 사용자가 정의한 단계 이름을 manifest의 `chapters[].custom_stage` 필드에 기록한다.

## Clean Language 룰의 적용

Custom framework에서도 Layer 1의 **Clean Language** 룰은 그대로 강제된다. 오히려 더 엄격하게 적용된다.

- 사용자가 framework 본문에 적은 *단계 이름*과 *질문 표현*을 AI가 그대로 사용한다. AI가 다른 단어로 바꿔치지 않는다.
- 예: 사용자가 *"S — Subjective: 환자가 자기 말로 증상을 설명한다"*라고 적었으면, AI는 인터뷰에서 *"Subjective 단계로 가볼까요. 환자가 자기 말로 증상을 설명한다고 하셨는데..."* 라고 묻는다. *"주관적 단계"*·*"환자 진술 단계"* 같은 동의어로 바꾸지 않는다.
- 사용자가 발화한 메타포(예: 환자를 *"손님"*이라 불렀다면)도 그대로 인터뷰 끝까지 따라간다.

이는 Custom framework에서 가장 중요한 룰이다. 사용자가 자기만의 활을 정의했다는 것은 *그 단어들이 의미를 가진다*는 신호이며, AI가 그 단어를 마음대로 추상화·추정하면 framework의 의도가 통째로 무너진다.

## 어울리는 앱 카테고리

**모두 (사용자 판단).** Custom은 카테고리 가중치와 무관하게 사용자가 자기 도메인에 적합하다고 판단할 때 선택한다. 다음 경우에 특히 의미가 있다:

- **`sensitive-regulated`** — 의료·금융·법무 등 *도메인 표준 절차*가 이미 존재하는 영역
- **`b2b-enterprise`** — 사용자(고객사)에게 이미 자기 워크플로 모델이 있는 경우
- **다국어·문화권 특화** — 한국어 사용자가 한국 시조 3장 구조나 풍수지리의 4국면 구조를 적용하고 싶을 때

기본 framework로 시작했다가 인터뷰 도중 *"이 활은 우리 앱에 안 맞다"*고 느껴지면 사용자는 언제든 Custom으로 전환할 수 있다. 다만 framework 전환은 L2 이하 산출물에 영향을 주므로 AI가 사용자에게 경고한 뒤 전환한다.

## 챕터 시드로의 변환 가이드

Custom framework의 챕터 시드는 *사용자가 정의한 N단계*가 그대로 N개 챕터 후보가 된다. 변환 가이드를 framework 본문에 적지 않았다면 AI는 다음과 같이 진행한다:

1. L2 인터뷰에서 N단계가 모두 채워진 직후 사용자에게 *"이 N단계를 그대로 N개 챕터로 만들까요, 합치거나 나누고 싶은 단계가 있나요?"* 묻는다.
2. 사용자가 합치고 싶다면 manifest의 `chapters[].custom_stage`에 배열로 기록한다 (예: `[subjective, objective]`).
3. 사용자가 나누고 싶다면 같은 단계를 여러 챕터로 분할하고 각 챕터에 *부제*를 붙인다 (예: `S-증상-기술` / `S-과거력`).

## 작성 후 다음 단계

1. 사용자가 `docs/spec/framework.md`를 작성한다.
2. AI가 그 파일을 읽어 *최소 요구 충족 여부*를 검증한다 (시작-끝 한 쌍 + N단계 정의).
3. 미충족 시 AI가 사용자에게 *어느 부분이 비었는지*만 알리고 보충을 기다린다. AI가 임의로 채우지 않는다.
4. 충족되면 manifest에 `framework: custom` 기록 + L1 인터뷰 시작.
