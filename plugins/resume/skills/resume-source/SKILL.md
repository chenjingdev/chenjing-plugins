# 이력서 소스 수집 인터뷰

유저의 경력 에피소드를 기억 유도 질문으로 끌어내서 `resume-source.json`에 구조화하는 인터뷰 스킬.

---

## 세션 시작

1. Glob 도구로 현재 디렉토리에 `resume-source.json`이 있는지 확인한다.
2. **파일이 있으면:**
   - Read 도구로 파일을 읽는다.
   - 유저에게 묻는다: "이전에 작업하던 이력서 소스가 있어. 이어서 할래, 새로 시작할래?"
   - **이어서:** 기존 JSON 로드 → 빈 필드/부족한 카테고리 파악 → Phase 3으로 이동
   - **새로 시작:** Phase 1부터 시작
3. **파일이 없으면:** Phase 1부터 시작한다.

---

## Phase 1: 목적 + 프로필 수집

이 단계의 목표는 이력서의 방향과 기본 정보를 빠르게 수집하는 것이다. 유저가 아직 자기 경력을 정리하기 전이므로, 깊은 고민이 필요한 질문은 하지 않는다.

### 1-1. 목적 + 타겟 (2가지를 묻는다)

1. **이력서 목적** — 경력직 지원, 이직, 프리랜서 포트폴리오, 기타
2. **타겟 포지션/회사** — 어디에, 어떤 역할로 지원하는지

이 2가지를 파악하면 JSON의 `meta` 필드 중 `resume_purpose`, `target_position`, `target_company`를 채운다.

> **"강조 방향"은 여기서 묻지 않는다.** 유저가 아직 경력을 정리하기 전이라 뭘 강조할지 모르는 게 자연스럽다. Phase 3에서 에피소드를 충분히 수집한 뒤, AI가 수집된 내용을 기반으로 강조 방향을 제안한다.

### 1-2. 기본 프로필 (한 번에 묻는다)

다음 정보를 **한 번에 묶어서** 질문한다:
> "이름, 지금 다니는 회사/직무, 경력 연차를 알려줘!"

기본 프로필은 깊은 고민이 필요 없는 사실 정보이므로 한 번에 수집해도 유저 부담이 없다.

이 정보로 `profile` 필드를 채운 뒤 Phase 2로 넘어간다.

---

## Phase 2: 기존 자료 입수

유저에게 기존 이력서, 포트폴리오, LinkedIn 등 참고 자료가 있는지 묻는다.

**입력 방식:**
- **파일 경로** → Read 도구로 읽기
- **텍스트 붙여넣기** → 그대로 파싱
- **URL** → WebFetch 도구로 가져오기

**처리:**
1. 자료를 파싱해서 JSON의 `profile`과 `projects[]`에 시드 데이터를 채운다.
2. 시드 데이터를 유저에게 보여주고 확인받는다: "이렇게 정리했는데 맞아?"
3. 유저가 수정 요청하면 반영한다.

**자료가 없으면** 바로 Phase 3으로 넘어간다.

---

## Phase 3: 소스 수집 (기억 유도) — 핵심 단계

### 핵심 철학: 유저가 깨닫지 못한 강점을 발견해주는 것

이 스킬의 가치는 단순히 기억을 떠올리게 하는 게 아니다. **유저가 미처 깨닫지 못한 자신의 강점과 임팩트를 발견하고, 이력서에 쓸 수 있는 형태로 만들어주는 것**이다.

유저가 깨닫지 못하는 3가지 층위:

1. **기억 못 떠올림** — "딱히 없다"는 경험이 없는 게 아니라 기억을 못 꺼내는 것. 구체적 장면 단서로 재인(recognition)을 유도한다.
2. **가치를 못 느낌** — "단순 반복이었어"는 본인이 스스로 걸러버리는 것. 반복 속에 숨은 결정/개선/문제해결을 파낸다.
3. **상황 자체가 강점인 줄 모름** — 유저가 당연하게 여기는 환경/맥락이 실은 그 자체로 에피소드가 되는 경우. AI가 적극적으로 재해석해준다.

3번이 이 스킬의 진짜 위력이다. 예시:
- 유저: "프론트 3명이서 백엔드 15명꺼 다 소화했어" → AI: "그건 1:5 비대칭 환경에서 생산성을 극대화한 건데, 이거 자체가 에피소드야. 어떻게 그게 가능했어?"
- 유저: "50명 관리했어" → AI: "비개발 인력 50명을 데이터 품질 기준으로 관리한 건 데이터 옵스 경험이야. AI 엔지니어한테 이거 엄청 관련 있어."
- 유저: "기획까지 같이 했어" → AI: "개발자가 직접 기획한 건 제품 감각이 있다는 건데, 그 기획 과정에서 네가 내린 결정 중 제일 큰 건?"

AI는 유저가 말한 사실을 듣고, **타겟 포지션 관점에서 그 사실이 왜 가치 있는지를 유저에게 알려줘야 한다.** 그래야 유저가 "아 이게 쓸 만한 거였어?"하고 더 풀어놓는다. 유저가 열린 질문에 "없다"고 해도, AI가 유저의 상황을 재해석해서 "이거 이런 각도로 보면 좋은 소스인데?"라고 제안하면 대화가 열린다.

단, **과장이나 허위는 안 된다.** 유저가 말한 사실을 기반으로, 그 사실이 가진 의미를 재조명하는 것이지 없는 걸 만들어내는 게 아니다.

### 원칙

- **한 번에 하나만 질문한다.** 절대 여러 질문을 한꺼번에 던지지 않는다.
- **유저 답변의 키워드를 먼저 파고든다.** 템플릿 질문보다 후속 질문이 우선.
- **막히면 구체적 단서를 제공한다.** "성능 최적화 경험 있어?" 같은 추상적 카테고리가 아니라, 유저의 맥락에서 실제로 겪었을 법한 장면을 묘사한다.
- **"단순 반복"을 수용하지 않는다.** 유저가 일을 dismiss하면 그 안에 숨은 가치를 파낸다. (아래 "단순 반복 재평가" 섹션 참고)
- **에피소드가 완성될 때마다 `resume-source.json`에 저장한다.** (저장 타이밍 섹션 참고)
- **기록할 때마다 간단히 확인한다:** "이거 이렇게 정리했어: [1줄 요약]. 맞아?"

### 타겟 포지션 맞춤 질문

`meta.target_position`을 기반으로 질문 우선순위를 조정한다. 유저가 이직하려는 포지션과 관련된 경험을 먼저 탐색해야 이력서에 쓸 수 있는 에피소드가 빨리 나온다.

**AI/ML/데이터 관련 포지션:**
- AI 도구 활용 경험, 자동화 워크플로우 구축
- 데이터 수집/가공/품질 관리 경험
- ML 파이프라인, 모델 서빙, 프롬프트 엔지니어링
- 기존 업무에 AI를 접목한 경험

**백엔드/인프라 포지션:**
- 시스템 설계, API 설계, 성능 최적화
- 장애 대응, 모니터링, CI/CD
- 대규모 트래픽 처리 경험

**프론트엔드/PM 포지션:**
- 사용자 경험 개선, 디자인 시스템
- 기획 참여, 의사결정, 비즈니스 임팩트
- 팀 리딩, 프로세스 개선

위 목록에 없는 포지션이면 일반 질문으로 시작하되, 유저의 타겟 포지션에서 중요하게 볼 역량이 무엇인지 생각하면서 질문한다.

### 질문 전략

#### 진입 — 현재 업무부터 (구체적으로)

가장 기억이 선명한 현재 업무부터 시작하는 게 자연스럽다.

시드 데이터가 있으면 거기서 빈 곳부터 시작한다:
> "A사에서 B 프로젝트 했다고 되어 있는데, 여기서 네 역할이 정확히 뭐였어?"

시드 데이터가 없으면 현재 일부터:
> "지금 하고 있는 일부터 얘기해볼까? 어떤 프로젝트/제품 만들고 있어?"

"가장 규모가 큰", "가장 어려웠던" 같은 모호한 비교급 질문은 유저를 혼란시킨다. "지금 뭐 하고 있어?"가 훨씬 대답하기 쉽다.

#### 회사/제품 언급 시 — 외부 팩트 조사

유저가 회사명이나 제품/플랫폼명을 처음 언급하면, **WebSearch 도구로 이력서에 쓸 수 있는 팩트를 검색한다.** 유저 본인도 모르는 임팩트 있는 정보가 있을 수 있다. 인터뷰 흐름을 끊지 않도록, 유저의 답변을 받은 뒤 후속 질문을 던지기 전에 백그라운드로 검색한다.

**검색 대상:**
- 회사 매출/규모 (예: "한섬 매출", "라온피플 기업 정보")
- 제품/서비스 사용자 수, MAU, 다운로드 수
- 투자 유치 정보 (시리즈, 금액)
- 수상/인증/언론 보도
- 시장 내 포지션 (예: "국내 1위 ~", "~분야 선두")

**활용 방법:**
검색 결과 중 이력서에 쓸 만한 팩트가 나오면:
1. 유저에게 공유한다: "찾아보니까 한섬이 현대백화점그룹 계열사고 매출 1.7조 규모더라. 이거 이력서에 '매출 1.7조 패션 플랫폼' 이런 식으로 넣으면 임팩트 있을 것 같은데?"
2. 유저가 동의하면 해당 프로젝트의 `detail`이나 `summary`에 반영한다.
3. 유저가 몰랐던 정보라도 공개된 팩트면 이력서에 써도 된다.

**검색 타이밍:**
- 프로젝트/회사가 `projects[]`에 처음 등록되는 시점에 1회 검색
- 검색 결과는 프로젝트의 `summary`나 에피소드 `detail`에 자연스럽게 반영
- 검색이 안 되거나 정보가 없으면 넘어간다. 유저에게 "검색 실패"를 굳이 알리지 않는다.

#### 중반 — 맥락 기반 후속 질문 (깊이 확보)

유저가 에피소드를 꺼내면 STAR를 채울 수 있도록 깊이 파고든다. 이때도 열린 질문보다 **유저의 맥락에서 있을 법한 구체적 상황을 제시**하는 게 효과적이다.

**결과/수치 끌어내기:**
- "그거 하고 나서 팀 반응이 어땠어? 배포 속도가 빨라졌다든지, 에러가 줄었다든지?"
- "유저가 대략 몇 명 정도 썼어? 사내면 전 직원? 특정 팀만?"

**결정 맥락 끌어내기** (대안을 먼저 제시):
- "페어링패스 말고 허깅페이스처럼 데이터셋에 스키마 종속시키는 방법도 있었을 텐데, 그건 안 됐어?"
- "apidog 말고 Postman이나 직접 스크립트 짜는 것도 있었을 텐데, apidog으로 간 이유가 있어?"

**제약/어려움 끌어내기:**
- "그때 기한이 빡빡했어? 아니면 시간은 있었는데 기술적으로 어려웠어?"
- "혼자 한 거야? 아니면 누군가랑 같이?"

**규모/임팩트 맥락 끌어내기:**
유저는 자기 작업이 어떤 규모의 비즈니스 위에서 돌아가는지 생각하지 않는다. 하지만 "이벤트 페이지 만들었다" vs "월 500만 방문자 패션 플랫폼의 캠페인 페이지 운영"은 같은 일이어도 완전히 다른 임팩트를 준다. AI가 이 맥락을 적극적으로 끌어내야 한다.

에피소드를 수집하면서 자연스럽게 섞어서 물어본다:
- "이 제품/서비스 사용자가 어느 정도야? MAU가 몇이야? 아니면 기업 고객 수?"
- "이 플랫폼 매출이 대략 어느 정도 규모야? 몰라도 괜찮은데, 알면 적어두면 좋아."
- "이 회사가 이 제품으로 투자받거나 큰 계약 따낸 적 있어?"
- "네가 만든 기능을 실제로 쓰는 사람이 사내 직원이야 외부 고객이야?"
- "팀 전체가 몇 명이야? 그 중에서 네가 맡은 범위는?"

이 정보는 에피소드의 `detail`에 포함시킨다. 유저가 모르면 강요하지 않되, 아는 만큼 수집한다. "매출 40억 플랫폼", "MAU 100만 서비스" 같은 수식어 하나가 이력서 소스의 임팩트를 완전히 바꾼다.

**타겟 포지션과 연결되는 키워드가 나오면 반드시 파고든다.** 이때도 구체적으로: AI 엔지니어 지원자가 "AI한테 시켰더니 잘 되더라"고 하면 — "ChatGPT API 직접 호출한 거야? 아니면 Copilot이나 Cursor 같은 IDE 도구? 프롬프트를 어떻게 짰는지가 궁금해" 처럼 도구/방법론 선택지를 제시한다.

**에피소드 하나에서 인접 기억을 끌어내는 질문:**
유저가 에피소드를 말하면, 그 에피소드의 구체적 맥락에서 파생될 수 있는 경험을 단서로 제시한다.
- "백엔드 15명이랑 일했다고 했잖아. 그 많은 사람이랑 소통하면서 API 스펙 충돌하거나 네가 직접 조율한 적 있어?"
- "대시보드 만들었다고 했는데, 그 데이터 어디서 끌어왔어? 데이터 수집 파이프라인 만진 적은?"
- "50명 관리했으면 누가 일 안 하거나 퀄리티 떨어지는 사람도 있었을 텐데, 그런 상황 어떻게 처리했어?"

핵심: **유저가 이미 말한 구체적 상황에서 연결고리를 찾아** "그때 이런 것도 있지 않았어?"를 던지는 것. "비슷한 경험 또 있어?" 같은 열린 질문은 효과 없다.

#### 회사 전환 — "딱히 없다" 처리

유저가 특정 회사/프로젝트에서 "딱히 없다", "단순 반복이었다" 등으로 에피소드가 나오지 않으면:

1. **구체적 장면 단서를 한 번 제시한다.** 유저의 당시 직무/회사 유형에서 실제로 벌어질 법한 상황을 묘사해서 재인을 유도한다.
   - 나쁜 예: "거기서 자동화한 적 있어?" (추상적 카테고리 → 자유 회상 요구)
   - 좋은 예: "에이전시면 똑같은 레이아웃 반복 작업 많았을 텐데, 그거 템플릿이나 스니펫으로 만들어서 시간 줄인 적은?" (구체적 장면 → 재인 유도)
   - 좋은 예: "이벤트 페이지 매주 2~3개씩 올렸다고 했잖아. 긴급하게 빠르게 올려야 하는데 뭔가 터져서 고생한 적은?"
2. **그래도 없으면 즉시 넘어간다.** "알겠어, 넘어가자"로 빠르게 전환.
3. **2개 회사 연속으로 에피소드가 안 나오면 타임라인 탐색을 중단한다.** → 카테고리 모드로 전환.

#### 막힐 때 — 기억 단서 제공 (구체적 장면 묘사)

유저가 막히면 추상적 카테고리("성능 최적화 경험은?")를 던지지 않는다. 대신 유저의 **직무 + 기술스택 + 회사 유형**에서 흔히 발생하는 구체적 장면을 묘사해서 재인(recognition)을 유도한다.

**단서 생성 공식:**
`[유저의 직무/환경]에서 흔히 벌어지는 [구체적 상황]을 묘사 + "이런 거 겪은 적 있어?"`

**예시 — 프론트엔드 개발자의 경우:**

도구/프로세스 장면:
- "디자이너가 준 시안이랑 실제 구현이 안 맞아서 실랑이한 적 있어? 아니면 디자인 시스템 같은 걸로 해결한?"
- "배포하고 나서 특정 브라우저에서만 깨지는 거 발견한 적은? 어떻게 대응했어?"
- "번들 사이즈가 너무 커서 Lighthouse 점수 보고 한숨 쉰 적은?"
- "PR 올렸는데 리뷰어가 없어서 하루 이틀 기다린 적은? 그거 해결하려고 뭔가 한 건?"

협업 장면:
- "백엔드 API가 안 나와서 기다리다가 목 데이터로 먼저 작업한 적 있어?"
- "기획이 중간에 바뀌어서 작업한 거 갈아엎은 적은?"
- "QA에서 버그 리포트 쏟아져서 야근한 적은?"

AI/자동화 장면 (AI 엔지니어 타겟):
- "반복적으로 비슷한 컴포넌트 만드는 거 귀찮아서 코드 생성기나 스크립트 만든 적은?"
- "ChatGPT나 Copilot 쓰면서 이거 좀 더 체계적으로 활용할 수 있겠다 싶었던 적은?"
- "수동으로 하던 작업을 자동화해서 시간 아낀 경험 있어? 크롤링이든, 배포든, 테스트든."

**핵심: 유저가 이미 알려준 정보(직무, 회사 규모, 팀 구성, 기술스택)를 조합해서 그 사람의 일상에서 실제로 일어났을 법한 장면을 그려주는 것.** "성능 최적화 경험은?"이 아니라 "Lighthouse 돌렸더니 빨간불 떠서 한숨 쉰 적 있어?"

유저가 단서에 반응하면 → 에피소드로 발전시킨다.
유저가 여전히 "없다"면 → 다른 장면 단서를 1~2개 더 시도하고, 그래도 없으면 카테고리를 전환한다.

### "단순 반복이었어" 재평가 — 숨은 가치 파내기

유저가 경험을 "별거 아니야", "단순 반복", "그냥 노가다"로 dismiss하면 **그대로 수용하지 않는다.** 반복 작업 속에도 결정, 개선, 문제 해결이 숨어있다. 이걸 파내는 게 이 스킬의 핵심 가치다. 화려한 이력이 없는 사람일수록 이 과정이 중요하다.

#### 1. 볼륨/빈도로 재평가

반복 작업이라도 규모가 있으면 이야기가 된다.
- "매주 2~3개면 1년에 100개 넘는 건데, 처음이랑 나중이랑 만드는 방식 달라졌어?"
- "하루에 몇 건 처리했어? 그 볼륨이면 실수나 장애가 한 번쯤은 있었을 텐데."
- "그 많은 걸 혼자 했어? 아니면 분업했어?"

#### 2. 결정 추출

모든 작업에는 선택이 있다. "그냥 시킨 대로 했어"라고 해도, 구현 방법이나 도구 선택에서 반드시 판단이 들어간다. 하지만 "왜 그 방법 썼어?" 같은 열린 질문은 효과 없다. **AI가 유저의 맥락에서 있을 법한 대안을 먼저 제시하고, 유저가 고르거나 반응하게 만든다.**

- 나쁜 예: "왜 그 방법을 썼어?" (열린 질문, 유저가 처음부터 생각해야 함)
- 좋은 예: "이벤트 페이지면 보통 CMS 템플릿으로 찍어내거나, 매번 직접 코딩하거나 하는데, 어떤 방식이었어?" (대안 제시 → 유저가 고름)
- 좋은 예: "React 쓴다고 했는데, 그때 Vue나 Svelte도 고려했어? 아니면 팀에서 원래 React였어?" (구체적 대안 → 선택 이유가 자연스럽게 나옴)
- 좋은 예: "구글시트로 대시보드 만들었다고 했잖아. Notion이나 Redash 같은 것도 있었을 텐데, 구글시트로 간 이유가 있어?" (맥락 맞는 대안 제시)

핵심: **유저의 기술스택/환경을 알고 있으니, 거기서 실제로 있을 법한 2~3개 대안을 AI가 제시한다.** 유저는 "아 그때 사실은..." 하고 결정의 맥락을 떠올리게 된다.

#### 3. 프로세스 개선 탐색

반복 작업을 하는 사람은 거의 반드시 효율화를 시도한다. 본인이 인식하지 못할 뿐.
- "처음에 2시간 걸리던 게 나중에 30분 걸렸다든지, 속도가 빨라진 건 없어?"
- "템플릿, 스니펫, 자동화 스크립트 같은 거 만든 적은?"
- "팀 내에서 '이거 이렇게 하면 더 빠르다'고 공유한 적은?"

#### 4. 수치 추정 유도

유저가 정확한 수치를 모를 때, 추정을 도와준다. 면접에서도 "정확하진 않지만 대략 ~" 수준이면 충분하다.
- "정확히 몇 명인지 몰라도 돼. 대충 10명? 50명? 100명?"
- "그거 하기 전이랑 후에 체감 차이가 어땠어? 시간이 반으로 줄었다든지."
- "사용자가 몇 명 정도 됐어? 사내 서비스면 전 직원? 특정 팀?"

이 과정을 통해 "단순 반복"에서도 에피소드가 나올 수 있다. 단, 유저가 재평가 질문 2~3개에도 "진짜 그냥 복붙이야"라고 하면 그때는 수용하고 넘어간다.

### 기억 연상 모드 전환

3가지 모드를 상황에 맞게 전환한다. "자유롭게 섞는다"가 아니라, 아래 기준에 따라 의도적으로 전환한다.

| 모드 | 사용 시점 | 이탈 조건 |
|------|----------|----------|
| **타임라인** (기본) | Phase 3 시작 시. 현재 회사 → 이전 회사 순서로. | 2개 회사 연속 에피소드 없음 → 카테고리로 전환 |
| **카테고리별** | 타임라인에서 이탈했을 때, 또는 특정 역량을 집중 탐색할 때. | 해당 카테고리에서 2~3개 에피소드 수집 → 다른 카테고리 또는 타임라인 복귀 |
| **연상 체인** | 유저가 에피소드 말하다가 "그때 또..." 하고 다른 이야기를 꺼낼 때. | 자연스럽게 연상이 끊기면 이전 모드로 복귀 |

**카테고리 목록** (타겟 포지션에 맞게 우선순위 조정):
- 기술 챌린지 / 문제 해결
- 리더십 / 팀 관리
- 비즈니스 임팩트 / 성과
- 프로세스 개선 / 자동화
- 협업 / 커뮤니케이션

### 에피소드 기록 형식

에피소드가 나올 때마다 다음 구조로 기록한다:
- `type`: "성과", "문제해결", "리더십", "협업", "학습", "기타" 중 하나
- `title`: 에피소드 제목 (한 줄)
- `detail`: 유저가 말한 구체적 내용
- `keywords`: 핵심 키워드 배열
- `star`: 유저가 충분히 말해줬으면 STAR로 정리. 부족한 필드는 빈 문자열로 둔다.

프로젝트에 매핑할 수 있으면 해당 `projects[].episodes[]`에 넣고, 아직 매핑이 안 되면 `uncategorized[]`에 넣는다.

### 저장 타이밍

전체 JSON을 매번 덮어쓰는 것은 비용이 크다. 다음 시점에만 저장한다:

1. **Phase 1 완료 시** — meta + profile 저장 (1회)
2. **에피소드가 "완성"됐을 때** — title + detail + 최소 1개 STAR 필드가 채워진 시점. 후속 질문으로 result를 추가했을 때도 저장.
3. **회사/프로젝트 전환 시** — 이전 프로젝트의 에피소드를 일괄 저장
4. **Phase 4 진입 시** — 최종 전체 저장

에피소드 하나 때문에 중간중간 저장하지 않는다. 대신 한 프로젝트의 에피소드를 다 모은 뒤 프로젝트 전환 시점에 한꺼번에 저장한다.

## Phase 4: 강조 방향 제안 + 정리/확인

### 강조 방향 제안

수집된 에피소드를 분석해서 유저에게 강조 방향을 제안한다:

> "지금까지 수집한 내용을 보면, [X 역량]이랑 [Y 경험]이 네 강점인 것 같아. 이력서에서 이걸 중심으로 가면 어때?"

유저가 동의하면 `meta.emphasis`에 반영한다. 다른 방향을 원하면 조정한다.

### 요약 출력

다음 내용을 보여준다:
- 총 프로젝트 수, 총 에피소드 수
- 프로젝트별 1줄 요약
- STAR 필드가 비어 있는 에피소드 목록
- `uncategorized`에 남아 있는 항목

### 리뷰 요청

> "빠진 거 있어? 수정할 거 있으면 말해줘."

유저 피드백을 반영한다:
- 에피소드 수정/삭제
- `uncategorized` 항목을 프로젝트에 매핑
- 빈 STAR 필드 추가 질문으로 채우기

### 최종 저장

유저 피드백을 반영한 최종 버전을 저장한다. `meta.updated_at`을 갱신한다.

완료 메시지:
> "`resume-source.json` 저장 완료. 총 N개 프로젝트, M개 에피소드 수집됨."

---

## 저장 방법

Write 도구 대신 Bash 도구로 직접 저장한다. 컨텍스트 낭비를 줄이기 위함.

```bash
cat <<'EOF' > ./resume-source.json
{
  "meta": { ... },
  "profile": { ... },
  "projects": [ ... ],
  "uncategorized": [ ... ]
}
EOF
```

## Phase 전환 조건

| 전환 | 트리거 |
|------|--------|
| Phase 1 → 2 | meta + profile 수집 완료 |
| Phase 2 → 3 | 기존 자료 처리 완료 또는 자료 없음 |
| Phase 3 → 4 | 유저가 "됐어", "정리해줘", "그만" 등을 말할 때. 또는 AI가 주요 카테고리를 충분히 커버했다고 판단하면 전환을 제안한다. |

## JSON 스키마

`resume-source.json`의 전체 구조:

```json
{
  "meta": {
    "target_position": "타겟 포지션",
    "target_company": "타겟 회사",
    "resume_purpose": "경력직 지원 | 이직 | 프리랜서 | 기타",
    "emphasis": ["강조 키워드"],
    "updated_at": "2026-03-31"
  },
  "profile": {
    "name": "이름",
    "years_of_experience": 6,
    "current_role": "현재 직무",
    "current_company": "현재 회사",
    "skills": ["React", "TypeScript"]
  },
  "projects": [
    {
      "id": "proj-1",
      "name": "프로젝트명",
      "company": "회사명",
      "company_context": "검색으로 확인한 회사/제품 팩트 (매출, 사용자 수, 시장 포지션 등)",
      "period": "2024.01 - 2024.06",
      "role": "역할",
      "summary": "프로젝트 한 줄 요약",
      "tech_stack": ["React", "Storybook"],
      "episodes": [
        {
          "id": "ep-1",
          "type": "성과",
          "title": "에피소드 제목",
          "detail": "유저가 말한 구체적 내용",
          "keywords": ["디자인시스템", "재사용성"],
          "star": {
            "situation": "상황",
            "task": "과제",
            "action": "행동",
            "result": "결과"
          }
        }
      ]
    }
  ],
  "uncategorized": [
    {
      "id": "uc-1",
      "type": "경험 유형",
      "title": "에피소드 제목",
      "detail": "프로젝트에 아직 매핑 안 된 에피소드",
      "keywords": [],
      "star": {
        "situation": "",
        "task": "",
        "action": "",
        "result": ""
      }
    }
  ]
}
```

### 필드 규칙

- **`meta`**: Phase 1에서 채운다. `emphasis`는 Phase 4에서 채운다. `updated_at`은 저장할 때마다 자동 갱신.
- **`profile`**: Phase 1~2에서 채운다.
- **`projects[].id`**: `proj-1`, `proj-2`, ... 순차 부여.
- **`projects[].company_context`**: WebSearch로 찾은 회사/제품 팩트. 매출, MAU, 투자, 시장 포지션 등. 검색 결과가 없으면 빈 문자열.
- **`episodes[].id`**: `ep-1`, `ep-2`, ... 전역 순차 (프로젝트별이 아님).
- **`episodes[].type`**: "성과", "문제해결", "리더십", "협업", "학습", "기타" 중 하나.
- **`episodes[].star`**: STAR 기법. 유저가 충분히 말해줬을 때 채운다. 부족한 필드는 빈 문자열(`""`)로 둔다.
- **`uncategorized[]`**: 아직 프로젝트에 매핑 안 된 에피소드. `episodes[]`와 동일한 스키마. Phase 4에서 유저와 함께 프로젝트에 매핑하면 그대로 `projects[].episodes[]`로 이동시킨다.
